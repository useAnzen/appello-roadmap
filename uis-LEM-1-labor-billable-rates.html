1. Business Case

Work Orders, T&M invoicing, and LEM sheet generation all require knowing what to bill the client for each trade classification at single time, time-and-a-half, and double time. Today, Employment Agreements only store cost rates (what the worker gets paid). Without structured billable rate data, Work Order line items would require manual fill-in-the-blank pricing — defeating the purpose of automation.

"We have their costs at single time, time and half, double time. But what do we bill them out at? So that way, if a work order is Corey Shelson and today I'm working as a journeyman, then it's going to pull my billable rate automatically." — Corey Shelson, March 30, 2026

Why This Must Come First

  • Work Order Line Items (UIS-20b) need to auto-populate billable rates when a trade level is selected
  • Equipment Invoicing (UIS-15B) establishes the cost-plus billing pattern that labor will follow
  • LEM Sheets require labor billing rates alongside equipment and material rates to generate complete reports
  • Without structured labor rates, "how do you build a work order if all your fields are just fill-in-the-blank fields with text?"

2. Current State Analysis

Existing Wage Rate Architecture

The Employment Agreement system already has a sophisticated rate management structure:

Model Role Key Fields
EmployeeAgreement Parent agreement (e.g., "UA Local 488") id, name
EmployeeAgreementContract Versioned contract with effective date id, effectiveDate, employeeAgreementId
EmployeeAgreementRateLabel Rate categories: Rate Types (isLineItem=false, e.g., "1x", "1.5x", "2x") and Line Items (isLineItem=true, e.g., "Base Wage", "Vacation Pay") id, name, isLineItem, isEmployerExpense, wageRegionId
TradeLevel Trade classifications (e.g., "Journeyman", "1st Year Apprentice") id, name, orderNo, hourlyWage
TradeLevelRate Junction: one per (Contract + TradeLevel + RateLabel) combination. Stores baseValue = total of all employer-expense line items. id, baseValue, tradeLevelId, employeeAgreementContractId, employeeAgreementRateLabelId
EmployeeAgreementRateLineItem Individual rate breakdown items per wage region id, baseValue, tradeLevelRateId, wageRegionId, employeeAgreementRateLabelId

What's Missing

The TradeLevelRate model has baseValue (the cost rate) but no field for what the client gets billed. This is the gap we're filling.

3. Data Model Changes

Schema Change: TradeLevelRate

Add a single field to the existing model:

// apps/api/prisma/schema.prisma
model TradeLevelRate {
    // ... existing fields ...
    baseValue                    Decimal        @db.Decimal(10, 2)
+   billableValue                Decimal?       @db.Decimal(10, 2)
    // ... existing relations ...
}
Field Type Nullable Default Notes
billableValue Decimal(10, 2) Yes null The per-hour billable rate for this trade level at this rate type (ST/OT/DT). Null means "not configured" — UI shows empty cell rather than $0.

Migration SQL

-- Add billable rate column to TradeLevelRate
ALTER TABLE `TradeLevelRate` ADD COLUMN `billableValue` DECIMAL(10, 2) NULL;

Why on TradeLevelRate (not a new table)?

  • There is a 1:1 relationship between a cost rate and its billable rate for a given (Contract + TradeLevel + RateType) combination
  • The existing initializeTradeLevelRates mutation already creates all the TradeLevelRate rows — the billableValue field is simply null until configured
  • The duplicateManyItemsForNewContract model method will automatically carry billableValue to new contracts with zero extra work
  • The editManyTradeLevelRates mutation already handles batch updates — we extend it to accept billableValue
  • Simpler query path for Work Order line items: one join to TradeLevelRate gets both cost and billable rate

The Billable Rates tab reuses the exact same query and mutation pattern as the Wage Rates tab. The only backend change is adding the field to the schema, type, and model.

5. UI Specification

New Component: BillableRates.tsx

Location: apps/frontend/src/components/DetailsPage/EmployeeContract/Tabs/BillableRates.tsx

Layout

Mirrors the Wage Rates tab structure but is simpler — no line item breakdown, just the summary rate per trade level alongside the editable billable rate.

Header Row

  • Wage Region selector — same CustomCombobox as Wage Rates tab
  • Search field — filter trade classifications by name
  • Edit button — toggles edit mode (only on active/future contracts)

Tab Group

Uses existing TabGroup component. One sub-tab per Rate Type (EmployeeAgreementRateLabel where isLineItem = false). These represent the wage rate multipliers — typical tabs: 1x, 1.5x, 2x.

Rate Table (per sub-tab)

Column Source Editable Notes
Row header (sticky left) — No Two rows: "Cost Rate" (read-only) and "Billable Rate" (editable)
Trade Level columns TradeLevel.name No Column headers, sorted by orderNo then name
Cost Rate cell TradeLevelRate.baseValue No Read-only, styled as muted/gray. Shows total employer cost per hour.
Billable Rate cell TradeLevelRate.billableValue Yes (in edit mode) CurrencyInput with 2 decimal places. Empty when null.
Margin indicator Computed: (billableValue - baseValue) / baseValue * 100 No Small badge showing markup %. Green when positive, red when negative. Only shown when both values exist.

Edit Mode Behavior

  • Click edit icon → all Billable Rate cells become editable CurrencyInput fields
  • ExitPageWarning footer bar appears (same pattern as Wage Rates)
  • Cancel → reverts all changes
  • Save → calls editManyTradeLevelRates with changed billableValue entries
  • No timesheet change impact dialog needed (billable rates don't retroactively affect timesheets — they're forward-looking for new Work Orders)

Copy From Default Behavior

  • When on a non-default wage region, "Update From Default" copies both baseValue and billableValue from the default region's TradeLevelRates
  • This requires updating the existing copyDefaultWageTable mutation to also copy billableValue

Permissions

  • Tab visibility: hasPermissions({ EmployeeAgreementContract: { view: true } }, permissions)
  • Edit access: only on active or future-dated contracts (same gate as Wage Rates edit button)

8. Acceptance Criteria

  • Billable Rates tab visible on Employee Agreement Contract detail page, positioned after Wage Rates tab
  • Sub-tabs match rate types — one tab per EmployeeAgreementRateLabel where isLineItem = false (e.g., 1x, 1.5x, 2x)
  • Cost Rate row displays read-only baseValue from TradeLevelRate (same value shown as total in Wage Rates tab)
  • Billable Rate row is editable in edit mode using CurrencyInput with 2 decimal places
  • Margin row auto-calculates markup percentage when both cost and billable values exist. Shows green for positive margin, red for negative.
  • Wage Region selector filters rates by region (same behavior as Wage Rates tab)
  • Trade Classification search filters columns by name
  • Save persists via editManyTradeLevelRates mutation — only changed billableValue entries are submitted
  • Null billableValue renders as empty cell, not $0.00 — distinguishing "not configured" from "free"
  • Copy From Default copies billableValue alongside baseValue when updating a non-default wage region
  • Contract duplication carries billableValue to new contracts (existing duplicateManyItemsForNewContract method)
  • Edit button only visible on active or future-dated contracts
  • ExitPageWarning appears when unsaved changes exist, with correct change count
  • Existing Wage Rates tab unaffected — no regression in cost rate management
  • Permission gated by EmployeeAgreementContract view permission
