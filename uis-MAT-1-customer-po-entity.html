1 · Business Case & Customer Evidence ▼

Purchase orders are the contractual backbone connecting estimates to invoicing. Today, the customer PO is a free-text string on the Job — no uniqueness, no multi-PO support, no detail page. When a job has multiple POs (one per change order), the system can’t track which PO applies to which invoice. This creates manual reconciliation work for every PM.

“We need to figure out what we’re doing with the job model refactor, because right now, purchase order is a piece of data in a table attached to the estimate.” — Internal Architecture Discussion (March 2026)
“If only one PO on a job, auto-populate in invoices; if multiple, prompt user to select.” — Corey Shelson (Feb 2025 design session)
“PO Number Management — dedicated Job Purchase Order entity, PO dropdown in invoices, migration to backfill existing PO strings.” — Jira Q21030-9356

2 · Scope ▼

IN SCOPE

  • PurchaseOrder and PurchaseOrderLineItem Prisma models
  • Const-based types: CustomerPO / SupplierPO; statuses: Draft / Issued / Sent / PartiallyReceived / FullyReceived / Closed
  • Data migration of existing Job.purchaseOrder strings to PurchaseOrder records
  • Auto-sequenced PO numbering: CPO-#### for customer, SPO-#### for supplier
  • PO detail page with tabs: Overview, Line Items, Invoices, Estimates, Files
  • PO tab on Company detail page
  • PO section on Job detail page
  • PO dropdown selector on Estimate and Invoice forms (replacing free-text input)
  • PurchaseOrderLineItem.total computed at query time (quantity × unitPrice)
  • Search Core indexing for PO lookup
  • Route configuration for /purchase-orders and /purchase-orders/[id]

OUT OF SCOPE

  • Supplier PO CRUD and PDF generation (deferred to MAT-5)
  • Receiving against PO (deferred to MAT-6)
  • PO approval workflow
  • PO-to-PO linking (e.g., amendment tracking)
  • Mobile PO management

4 · Data Model ▼

Const-Based Type & Status Definitions

// PurchaseOrder types (const-based, not enum)
export const PurchaseOrderType = {
    CustomerPO: "CustomerPO",
    SupplierPO: "SupplierPO"
} as const;

// PurchaseOrder statuses
export const PurchaseOrderStatus = {
    Draft: "Draft",
    Issued: "Issued",
    Sent: "Sent",
    PartiallyReceived: "PartiallyReceived",
    FullyReceived: "FullyReceived",
    Closed: "Closed"
} as const;

Prisma Schema — PurchaseOrder

model PurchaseOrder {
    id              String    @id @default(cuid()) @db.VarChar(21)
    instanceId      String    @db.VarChar(21)
    type            String    @db.VarChar(30)   // CustomerPO | SupplierPO
    poNumber        String    @db.VarChar(50)
    companyId       String?   @db.VarChar(21)
    jobId           String?   @db.VarChar(21)
    contactId       String?   @db.VarChar(21)
    status          String    @default("Draft") @db.VarChar(30)
    issuedDate      DateTime?
    amount          Decimal?  @db.Decimal(10, 2)
    description     String?   @db.Text
    terms           String?   @db.VarChar(191)
    isArchived      Boolean   @default(false)
    isDeleted       Boolean   @default(false)
    createdAt       DateTime  @default(now())
    updatedAt       DateTime  @updatedAt
    createdById     String?   @db.VarChar(21)
    updatedById     String?   @db.VarChar(21)

    Instance        Instance  @relation(fields: [instanceId], references: [id])
    Company         Company?  @relation(fields: [companyId], references: [id])
    Job             Job?      @relation(fields: [jobId], references: [id])
    Contact         Contact?  @relation(fields: [contactId], references: [id])
    CreatedBy       User?     @relation("PurchaseOrderCreatedBy", fields: [createdById], references: [id])
    UpdatedBy       User?     @relation("PurchaseOrderUpdatedBy", fields: [updatedById], references: [id])
    LineItems       PurchaseOrderLineItem[]

    @@unique([instanceId, poNumber])
    @@index([instanceId])
    @@index([jobId])
    @@index([companyId])
    @@index([type])
    @@index([status])
}

Prisma Schema — PurchaseOrderLineItem

model PurchaseOrderLineItem {
    id                String         @id @default(cuid()) @db.VarChar(21)
    purchaseOrderId   String         @db.VarChar(21)
    description       String?        @db.Text
    quantity          Decimal?       @db.Decimal(10, 4)
    unitPrice         Decimal?       @db.Decimal(10, 2)
    accountItemId     String?        @db.VarChar(21)
    costCodeId        String?        @db.VarChar(21)
    sortOrder         Int            @default(0)
    isDeleted         Boolean        @default(false)
    createdAt         DateTime       @default(now())
    updatedAt         DateTime       @updatedAt

    PurchaseOrder     PurchaseOrder  @relation(fields: [purchaseOrderId], references: [id])
    AccountItem       AccountItem?   @relation(fields: [accountItemId], references: [id])
    CostCode          CostCode?      @relation(fields: [costCodeId], references: [id])

    @@index([purchaseOrderId])
}

Data Migration SQL

-- Migrate existing Job.purchaseOrder strings to PurchaseOrder records
INSERT INTO PurchaseOrder (
    id, instanceId, type, poNumber, jobId, status,
    issuedDate, createdAt, updatedAt
)
SELECT
    SUBSTRING(MD5(CONCAT(j.id, '-po')), 1, 21),
    j.instanceId,
    'CustomerPO',
    j.purchaseOrder,
    j.id,
    'Issued',
    j.createdAt,
    NOW(),
    NOW()
FROM Job j
WHERE j.purchaseOrder IS NOT NULL
  AND j.purchaseOrder != ''
  AND j.isDeleted = 0;

7 · Acceptance Criteria ▼

  • PurchaseOrder and PurchaseOrderLineItem tables created with indexes
  • Existing Job.purchaseOrder strings migrated to PurchaseOrder records with type=CustomerPO, status=Issued
  • Multiple POs per job supported — Job detail shows all linked POs
  • PO detail page accessible at /purchase-orders/[id] with Overview, Line Items, Invoices, Estimates, Files tabs
  • PO tab on Company detail page shows all POs for that company
  • CPO-#### auto-sequenced numbering per instance
  • Estimates and Invoices have PO dropdown selector replacing free-text input
  • When job has single PO, auto-selected in invoice form
  • When job has multiple POs, user prompted to select
  • PurchaseOrderLineItem.total computed as quantity × unitPrice (not stored)
  • Search Core indexes POs for global search
  • No regression on existing estimate/invoice PO references after migration
  • Permission: users need appropriate permission to view/create/edit POs
  • Validation: poNumber required, type must be valid const value
  • Notification: none (POs are internal documents at this stage)
