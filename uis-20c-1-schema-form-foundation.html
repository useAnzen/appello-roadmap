The original spec included markupPercent and overheadPercent on the Job model for WO pricing. These are typically contract-level settings, not per-WO settings. Options:

3. Signature Model (Polymorphic)

Reusable signature model that can be tied to any record. Leverages the same signature canvas component (SignatureQuestionType from @useanzen/ui) used in SafetyForm submissions.

model Signature {
  id                String    @id @db.VarChar(21)
  isDeleted         Boolean   @default(false)
  isArchived        Boolean   @default(false)
  createdAt         DateTime  @default(now())
  updatedAt         DateTime  @default(now()) @updatedAt

  // Polymorphic reference
  referenceModel    String              // "Job" (for WOs), "FormSubmission" (future), etc.
  referenceId       String    @db.VarChar(21)

  // Signature data
  assetId           String    @db.VarChar(21)
  Asset             Asset     @relation(fields: [assetId], references: [id])

  userId            String?   @db.VarChar(21)
  User              User?     @relation(fields: [userId], references: [id])
  signerName        String?   @db.VarChar(191)  // for external signers not in the system
  signerEmail       String?   @db.VarChar(191)  // optional contact email
  signerCompany     String?   @db.VarChar(191)  // optional company affiliation

  signedAt          DateTime  @default(now())
  latitude          Decimal?  @db.Decimal(10, 7)
  longitude         Decimal?  @db.Decimal(10, 7)

  @@index([referenceModel, referenceId, isDeleted])
  @@index([assetId])
}

How it works with Work Orders

  • referenceModel = "Job", referenceId = jobId — multiple signatures per WO
  • The signature canvas component saves the drawn signature as an Asset (image), then creates a Signature record linking it to the WO
  • userId is set when the signer is a known user in the system; null for external signers (client representatives)
  • GPS coordinates captured at the moment of signing (browser geolocation API)
  • The signature section on the WO form is rendered via a DAA position (see Section 4)
Future reuse: The polymorphic pattern means signatures can be attached to any model. When SafetyForms migrate to this model (future), the referenceModel = "FormSubmission" pattern works identically. No schema changes needed.

4. DocumentAttributeAssignment Setup

WO-specific fields are positioned on the form via DocumentAttributeAssignments with referenceModel = "Job" and subModel containing the WorkOrder classification. This uses the same pattern as Expense forms where subModel = "EmployeeExpense" or "AccountPayable" filters which fields appear.

Step 1: Duplicate shared Job DAAs for Work Orders

Important: No comma-delimited subModel values. Investigation confirmed that subModel must be single individual values across the codebase. TableFeed uses exact equality matching, the unique constraint @@unique([referenceModel, subModel, slug]) requires single values, and the Expense system explicitly migrated AWAY from comma-delimited values into separate records. We follow the same pattern: duplicate shared fields into both "ProjectJob" and "WorkOrder" sets.
subModel ValueMeaning
NULL (existing)Current Job DAAs — updated to ProjectJob in the backfill
ProjectJobProject Job forms/tables/settings only
WorkOrderWork Order forms/tables/settings only

Migration approach (follows the Expense separation pattern)

  1. Backfill: Set all existing Job DAAs (currently subModel = NULL) to subModel = "ProjectJob"
  2. Duplicate shared fields: For fields that must also appear on WO forms, create a second copy of the DAA with subModel = "WorkOrder". Same defaultAttribute, same config, different subModel and sectionId (pointing to the WO section).
  3. Create WO-only fields: New DAAs for dateOfWork, labor items, material items, equipment items, signatures — only subModel = "WorkOrder"

Shared fields to duplicate for Work Orders

These existing Job DAAs need a duplicate record with subModel = "WorkOrder":

  • job-number — used for WO-001 numbering
  • job-name — always required
  • job-description — optional but useful context on WOs
  • job-rate-type — rate type context for the job
  • job-wage-region — critical: determines which WageRegion's TradeLevelRates are loaded for labor line items in WO-3b
  • job-purchase-order — customer reference (from PR #6831 PurchaseOrder config)
  • job-status — always needed (Draft, Submitted, etc.)

The original DAA records retain subModel = "ProjectJob". The duplicates get subModel = "WorkOrder" and are assigned to the appropriate WO DocumentSection.

Form subModel selection

The Job form (DataModelForm from PR #6831) needs to pass the correct subModel based on the job's classification:

  • When creating/editing a WorkOrder job: pass subModel = "WorkOrder"
  • When creating/editing a ProjectJob job: pass subModel = "ProjectJob"
  • This ensures only the correct set of DAAs loads for each form type

Job Type Dropdown Filtering by Classification

The jobType dropdown on the Job form must only show types whose jobClassification matches the current context. This prevents a user from accidentally selecting a WorkOrder type on a ProjectJob form or vice versa.

  • Create from Jobs page: the jobType dropdown shows only PropertyValues where jobClassification = "ProjectJob"
  • Create from Work Orders page: the jobType dropdown shows only PropertyValues where jobClassification = "WorkOrder"
  • Edit: the jobType dropdown is filtered to types matching the job's current classification (derived from its existing jobTypeId)
  • A job cannot be switched from ProjectJob to WorkOrder (or vice versa) via the form — the dropdown simply does not include cross-classification options

The backend guard for this constraint lives in WO-1 (Step 5, beforeEdit hook on the Job model), which rejects cross-classification changes at the API level. This frontend filtering is the UX layer that prevents invalid options from appearing.

The Job form uses DocumentWrapperForm with a renderChild callback. The switch statement on assignment.defaultAttribute determines which custom component renders at each DAA position:

    switch (assignment.defaultAttribute) {
        case "wo-date-of-work":
            return <FormControl type="date" ... />;
        case "wo-labor-items":
            return <LaborItemEditor ... />;        // WO-3b
        case "wo-material-items":
            return <MaterialItemEditor ... />;      // WO-3b
        case "wo-equipment-items":
            return <EquipmentItemEditor ... />;     // WO-3b

            return <SignatureSection ... />;         // WO-3c

        default:
            return <AttachableDocumentAttributeForm ... />;

5. Classification-Aware Job Number Generation

The existing Job.generateJobNumber() method (in apps/api/src/models/Job.ts, lines 1122-1197) generates numbers based on the project's numbering pattern. For Work Orders without a project, this needs updating.

Current behavior

  • Fetches all jobs for the project
  • Extracts prefix/postfix pattern from the most recent job number
  • Increments the numeric portion with padding from instance settings
  • fullJobNumber = ${project.number}-${number}

Updated behavior for Work Orders

  • Branching is based on jobClassification (derived from jobTypeId), NOT on whether a project is attached
  • When jobClassification = WorkOrder: use WO-specific prefix (default: "WO"), auto-increment across all WorkOrder jobs for the instance
  • fullJobNumber = "WO-001", "WO-002", etc. — regardless of project attachment
  • When jobClassification = ProjectJob: existing numbering logic from PR #6831 (handles both with-project and without-project cases)
Implementation: Extends the changes from PR #6831. Add a getJobClassification(jobTypeId) helper. The classification check goes at the top of generateJobNumber() — WorkOrder takes the WO numbering path, ProjectJob falls through to the existing logic. A WorkOrder under a project still gets WO-prefixed numbering; the project context is shown elsewhere in the UI.

6. Important Architecture Notes

DataModelForms — everything flows through the DataModel

The Job form was updated in PR #6831 to use DocumentWrapperForm. All WO fields are positioned via DAAs and rendered via the renderChild callback. The custom input components (LaborItemEditor, etc.) are just components returned by the switch statement — they don't have their own save logic.

Save is transactional — form-level, not field-level

Line item editors, signature sections, and all other custom inputs maintain local state. Nothing saves to the database until the user clicks Save Draft or Submit. On save, the form collects all data (job fields + line items + signatures) and submits in a single transaction. This matches the existing Job form pattern where editJob + upsertWorkOrderLineItems + signature creation happen in sequence.

Computed subtotals are display-only

The wo-labor-subtotal, wo-material-subtotal, and wo-equipment-subtotal DAAs render computed values from the line item editors' local state. They recalculate as the user adds/edits line items but don't save to the database. They're visual feedback for the user — the actual financial calculations happen at invoice generation (WO-7).

Subtotal display and viewWageDetails permission

The computed subtotals use rates from TradeLevelRate.billableValue (labor) and LEM-2 resolveEquipmentRate (equipment). These should only be shown if the user has user___viewWageDetails permission. On the form, if the user doesn't have this permission, the subtotal DAAs render as "— " or are hidden entirely.
