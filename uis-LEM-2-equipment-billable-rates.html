Equipment Billable Rates

Category-level rate configuration with per-equipment and per-job overrides using a polymorphic rate value table, editable rate types, and a rate resolution engine. Extracted from and replaces the data model layer of UIS-15C.

1. Business Case

Equipment billing in construction requires structured rate data at multiple levels. A CAT 320 excavator might bill $150/hr standard, $225/hr overtime, $75/hr standby. These rates differ by equipment category (all excavators vs. all skid steers), by individual machine (newer units command higher rates), and by job (negotiated project rates). Without structured rate data, Work Orders and LEM sheets require manual fill-in pricing.

"In construction, they call it labor equipment material, LEM. Right now, we don't have structured data for labor rates. We don't have structured data for material rates or equipment rates. It's like we kind of have to add all three of these for work orders to be functional." — Corey Shelson, March 30, 2026

Industry Context

  • Blue Book (EquipmentWatch) publishes Operating and Standby rates at hourly, daily, weekly, and monthly periods
  • FEMA 325 explicitly separates equipment rate and operator rate into two columns
  • Every major construction ERP (HCSS, B2W, CMiC, Viewpoint) supports category-level defaults with per-equipment and per-job overrides
  • Operator-included vs. bare (dry hire) is a fundamental distinction in every equipment billing system

2. Scope & Relationship to UIS-15C

What LEM-2 Covers (extracted from UIS-15C)

  • IN EquipmentRateLabel model — editable rate type definitions per EquipmentCategory
  • IN EquipmentRateSchedule model — versioned rate schedule per category with effectiveDate
  • IN EquipmentRateValue model — polymorphic rate values scoped via referenceModel + referenceId, linked to schedule
  • IN EquipmentRatePer enum — HOUR, DAY, WEEK, MONTH, YEAR
  • IN Operator-included flag on rate values
  • IN Rate resolution engine with cascading priority and provenance
  • IN UI: Rates tab on EquipmentCategory detail page
  • IN UI: Rates tab on Equipment detail page (inherited + override)
  • IN UI: Job rate overrides on Equipment detail (from equipment side)
  • IN UI: Equipment rate overrides on Job detail (from job side)

What Stays in UIS-15C (built on top of LEM-2)

  • UIS-15C Time-of-day rate windows & hour splitting across windows
  • UIS-15C Effective dating / rate schedule versioning
  • UIS-15C Attachment-specific pricing
  • UIS-15C Overtime multiplier auto-calculation

Out of Scope (designed for, not built)

  • FUTURE Customer-specific pricing — just add referenceModel = "Company", zero schema changes
  • FUTURE Project-level pricing — referenceModel = "Project"
  • FUTURE Unit-based rates (per cubic yard, per ton), mob/demob fees, fuel surcharges

4. Rate Hierarchy & Resolution

All rate values live in a single EquipmentRateValue table, scoped by referenceModel + referenceId. Resolution checks from most specific to least specific — first match wins.

4 Company (Customer) referenceModel = "Company" — contract pricing FUTURE
3 Job referenceModel = "Job" — negotiated project rates
2 Equipment referenceModel = "Equipment" — per-machine override
1 EquipmentCategory referenceModel = "EquipmentCategory" — fleet-wide fallback

Resolution Function

resolveEquipmentRate(equipmentId, rateLabelId, per, jobId?, operatorIncluded?, date?):
  equipment = Equipment.find(equipmentId)
  priority = ["Job", "Equipment", "EquipmentCategory"]

  // Determine active schedule for this category (if date provided)
  scheduleId = null
  if date:
    activeSchedule = EquipmentRateSchedule.findFirst({
      equipmentCategoryId: equipment.equipmentCategoryId,
      effectiveDate: { lte: date },
      orderBy: { effectiveDate: "desc" }
    })
    scheduleId = activeSchedule?.id ?? null

  candidates = EquipmentRateValue.findMany({
    equipmentRateLabelId: rateLabelId,
    per: per,
    operatorIncluded: operatorIncluded ?? true,
    equipmentRateScheduleId: scheduleId,  // null = default rates
    OR: [
      { referenceModel: "Job",               referenceId: jobId },
      { referenceModel: "Equipment",          referenceId: equipmentId },
      { referenceModel: "EquipmentCategory",  referenceId: equipment.equipmentCategoryId }
    ]
  })

  // Pick highest priority match
  for model in priority:
    match = candidates.find(c => c.referenceModel === model)
    if match: return { amount, per, operatorIncluded, provenance: model, scheduleId }

  return null

The per parameter is required — determined by what unit the user selects when adding equipment to a Work Order (e.g., "3 hours" → HOUR, "2 days" → DAY).

Polymorphic advantage: Adding Customer-level overrides in the future means adding referenceModel = "Company" rows and prepending "Company" to the priority array. Zero schema changes, zero migrations.

New Model: EquipmentRateSchedule

Groups EquipmentRateValue records into versioned rate schedules per category. Mirrors the EmployeeAgreementContract pattern for labor rate versioning. In LEM-2, the initial schedule represents the "current rates." UIS-15C later adds the ability to create future-dated schedules.

  id                    String              @id @db.VarChar(21)
  isDeleted             Boolean             @default(false)
  isArchived            Boolean             @default(false)
  createdAt             DateTime            @default(now())
  updatedAt             DateTime            @default(now()) @updatedAt
  name                  String              // e.g. "2026 Rates", "2027 Rate Increase"
  effectiveDate         DateTime            @db.Date
  description           String?             @db.Text

  equipmentCategoryId   String              @db.VarChar(21)
  EquipmentCategory     EquipmentCategory   @relation(fields: [equipmentCategoryId], references: [id])

  EquipmentRateValues   EquipmentRateValue[]

  @@index([equipmentCategoryId, effectiveDate, isArchived, isDeleted])
}

New Model: EquipmentRateValue (Polymorphic)

Single table for all rate values across all scope levels. The referenceModel + referenceId pair determines what the rate is scoped to. The optional equipmentRateScheduleId links values to a versioned schedule.

model EquipmentRateValue {
  id                    String              @id @db.VarChar(21)
  isDeleted             Boolean             @default(false)
  isArchived            Boolean             @default(false)
  createdAt             DateTime            @default(now())
  updatedAt             DateTime            @default(now()) @updatedAt

  // Polymorphic scope
  referenceModel        String              // "EquipmentCategory" | "Equipment" | "Job"
  referenceId           String              @db.VarChar(21)

  // Schedule versioning
  equipmentRateScheduleId String?            @db.VarChar(21)
  EquipmentRateSchedule   EquipmentRateSchedule? @relation(fields: [equipmentRateScheduleId], references: [id])

  // Rate data
  equipmentRateLabelId  String              @db.VarChar(21)
  EquipmentRateLabel    EquipmentRateLabel  @relation(fields: [equipmentRateLabelId], references: [id])
  amount                Decimal             @db.Decimal(12, 2)
  per                   EquipmentRatePer    @default(HOUR)
  operatorIncluded      Boolean             @default(true)

  @@unique([equipmentRateLabelId, referenceModel, referenceId, per, operatorIncluded, equipmentRateScheduleId])
  @@index([referenceModel, referenceId, isArchived, isDeleted])
  @@index([equipmentRateLabelId, referenceModel])
  @@index([equipmentRateScheduleId])
}

Future-Proofing

Adding a new scope level (e.g., Customer, Project) requires:

  1. Insert EquipmentRateValue rows with the new referenceModel value
  2. Add the new model name to the priority array in the resolution function
  3. Build the UI entry point for that scope

Zero schema changes. Zero migrations.

6. Operator-Included & Work Order Tie-In

How Operator-Included Works

operatorIncluded is a flag on the rate value indicating which billing scenario the rate applies to:

  • Operated (wet hire) — operatorIncluded = true — rate covers both machine and operator. The $165/hr rate includes the operator's cost.
  • Bare (dry hire) — operatorIncluded = false — rate covers machine only. Operator labor is tracked separately via labor line items (LEM-1 billable rates).

UI Behavior

  • The rates table shows one set of rates by default (operator-included = true, the more common scenario)
  • A toggle at the top of the table: "Also configure bare (equipment-only) rates" — reveals a second row per rate label for the without-operator rate
  • Most companies will only configure one mode. Companies that do both (e.g., crane rental) opt in to the second set.
  • Standby rates follow the same pattern — a Standby rate with operatorIncluded = true means "operator on-site waiting, equipment idle, rate covers both"

Work Order Tie-In

When logging equipment on a Work Order (UIS-20b line items) or equipment timesheet (UIS-15H):

  1. User selects equipment + rate type (Standard, OT, Standby, etc.)
  2. System calls resolveEquipmentRate(equipmentId, rateLabelId, jobId, operatorIncluded)
  3. Resolved rate auto-populates the line item amount
  4. If operator-included: the operator's labor is bundled in the equipment rate. No separate labor line item for that person's time on that equipment.
  5. If bare: the operator's labor is logged separately via labor line items using their trade level billable rate (from LEM-1). Both lines appear on the WO/LEM sheet.
LEM Sheet connection: A complete LEM (Labor, Equipment, Material) sheet draws from all three LEM rate systems: labor billable rates (LEM-1) for worker time, equipment billable rates (LEM-2) for machines, and material rates (LEM-3) for supplies. The operatorIncluded flag determines whether operator cost appears in the labor section or the equipment section.

8. UI Specification

8.1 Shared Component: EquipmentRateTable

A single reusable component used across all three entry points (EquipmentCategory, Equipment detail, Job detail). It takes referenceModel and referenceId as props to determine what scope it's saving to.

EquipmentRateTable
  Props:
    referenceModel: "EquipmentCategory" | "Equipment" | "Job"
    referenceId: string
    equipmentCategoryId: string   // to load rate labels
    parentRates?: EquipmentRateValue[]  // inherited values to show read-only (optional)

  Structure:
    Level 1 TabGroup: "Operator Included" | "Equipment Only"
      Level 2 TabGroup: "Hourly" | "Daily" | "Weekly" | "Monthly" | "Yearly"
        Table:
          Rows = EquipmentRateLabels (Standard, Standby, OT, Weekend, Holiday)
          Column(s):
            - If parentRates provided: "Default" (read-only) | "Override" (editable)
            - If no parentRates: single "Amount" column (editable)
          Edit mode + ExitPageWarning footer

Tab Behavior

  • Level 1 tabs filter by operatorIncluded — "Operator Included" = true, "Equipment Only" = false
  • Level 2 tabs filter by per — one tab per EquipmentRatePer value (Hourly, Daily, Weekly, Monthly, Yearly)
  • Not every tab needs values — a company might only configure Hourly and Daily rates. Empty tabs show rate labels with blank amount cells.
  • "Equipment Only" tab is hidden by default. A toggle or setting reveals it for companies that need bare rates.

Table Per Tab

  • Rows = all EquipmentRateLabel records for the category, ordered by orderNo
  • Each cell is a CurrencyInput — fill in the dollar value, leave blank for unconfigured
  • On Category page: single "Amount" column per row — these are the fleet defaults
  • On Equipment page: "Category Default" (read-only gray) + "This Equipment" (editable or "Inherited" badge) columns
  • On Job page: "Effective Rate" (resolved, read-only with provenance badge) + "Job Override" (editable or "Using default") columns
  • Save calls upsertManyEquipmentRateValues with the component's referenceModel and referenceId

8.2 Equipment Category — Rates Tab

Uses EquipmentRateTable with referenceModel="EquipmentCategory" and referenceId={categoryId}. No parentRates prop (this IS the base level).

  • Above the rate table: Rate Labels management section (add/edit/reorder/delete rate types)
  • Below: the EquipmentRateTable with the two-level tabs
  • Default labels (Standard, Standby, OT, Weekend, Holiday) seeded on first visit via initializeEquipmentRateLabels

8.3 Equipment Detail — Rates Tab

Uses EquipmentRateTable with referenceModel="Equipment", referenceId={equipmentId}, and parentRates loaded from the equipment's category.

  • Shows "Category Default" read-only column alongside the editable "This Equipment" column
  • "Inherited" badge when no equipment-level override exists for that (label + per + operatorIncluded) combination
  • "Reset" action per row deletes the equipment override
  • Below the equipment rate table: Job Rate Overrides section with Job selector (CustomCombobox) and a second EquipmentRateTable instance with referenceModel="Job"

8.4 Job Detail — Equipment Rate Overrides

New tab or section on Job detail page. Equipment selector (CustomCombobox) at the top, then EquipmentRateTable with referenceModel="Job" and parentRates loaded from the resolved equipment + category cascade.

  • "Effective Rate" column shows the resolved value with provenance badge (Category / Equipment / Job)
  • "Job Override" column is editable — "Using default" when no job override exists
  • "Clear" action per row removes the job override
  • This is where a PM sets job-specific rates before work begins — auto-populates WO line items
One component, three contexts: EquipmentRateTable renders the same two-level tab structure everywhere. The referenceModel prop determines what it saves to. The parentRates prop determines whether it shows an "inherited/default" read-only column alongside the editable values.

11. Acceptance Criteria

  • EquipmentRateLabel CRUD — create, edit, reorder, delete rate types per category
  • Default labels seeded — new categories get Standard, Standby, Overtime, Weekend, Holiday
  • Category rates configurable — amount, per (HOUR/DAY/WEEK/MONTH/YEAR), operatorIncluded
  • Equipment inherits category rates — "Inherited" badge for un-overridden labels
  • Equipment override — editing creates EquipmentRateValue with referenceModel = "Equipment"; "Reset" deletes it
  • Job override from Equipment detail — can set per-job rates; "Clear" deletes the override
  • Job override from Job detail — same data accessible from the job side with equipment selector and provenance badges
  • Rate resolution — Job > Equipment > EquipmentCategory cascade with provenance
  • resolveEquipmentRate query returns amount, per, operatorIncluded, provenance, rateLabelName
  • Polymorphic model works — all rate values in single EquipmentRateValue table using referenceModel + referenceId
  • Unique constraints enforced — no duplicate (rateLabelId + referenceModel + referenceId + per + operatorIncluded)
  • Two-level tab structure — Level 1: Operator Included / Equipment Only; Level 2: Hourly / Daily / Weekly / Monthly / Yearly
  • Multiple per-type rates — same rate label can have different amounts at different per-units (hourly ≠ daily ÷ 8)
  • Shared EquipmentRateTable component — one component used on Category, Equipment, and Job pages via referenceModel/referenceId props
  • Tabs visible — Rates tab on EquipmentCategory, Equipment detail, and Job detail pages
  • No regression — existing equipment CRUD, assignments, and other tabs unaffected

12. Dependencies

Requires

None — Can be built in parallel with LEM-1. No blocking dependencies.

Enables

UIS-15H Equipment Timesheet Capture — resolved rates at approval time
UIS-15C Equipment Variable Rates (advanced) — time windows, effective dating, attachments build on this
UIS-15A Equipment Job P&L — cost rollups use resolved rates
UIS-15B Equipment Invoicing — invoice lines use resolved billable rates
UIS-20b WO Line Item Tables — equipment line items auto-populate from resolved rates

Relationship to UIS-15C

LEM-2 extracts the data model and rate resolution engine from UIS-15C. After LEM-2 ships, UIS-15C becomes an enhancement layer adding time-of-day windows, effective dating, attachment pricing, and overtime multiplier auto-calculation on top of the LEM-2 foundation.
