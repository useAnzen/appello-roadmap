WO-5 Work Orders Draft

T&M Work Order Log

Work Orders list page at /work-orders. Extracts the Jobs list into a shared component (SharedJobListView) and reuses it with subModel = "WorkOrder". No new SQL views, no new backend — just a frontend refactor to share the existing ListTableViews infrastructure.

Estimate: 2-3 days Requires: WO-1 (JobClassification, WO statuses, Board + tableConfiguration)

1. Approach: Extract & Reuse

WO-5 is sequenced immediately after WO-1 as the first frontend work item — it gives Work Orders a visible home in the app.

The existing Jobs list page (pages/jobs/index.tsx) has everything we need: ListTableViews with TableFeed, Kanban, status updates, CRUD actions, search, filters, CSV export. Rather than duplicating, we extract it into a shared component parameterized by subModel.

In-Scope Deliverables

  1. Extract shared Job list logic from pages/jobs/index.tsx into SharedListViews/SharedJobListView.tsx
  2. Parameterize by subModel, status propertyName, labels, routes, create behavior
  3. Use on Jobs page with subModel = "ProjectJob" and statusPropertyName = "job status"
  4. Use on Work Orders page with subModel = "WorkOrder" and statusPropertyName = "workOrderStatus"
  5. Sidebar navigation — add a "Work Orders" entry in the sidebar linking to /work-orders
No backend changes. TableFeed already supports subModel filtering via configureColumns.ts (set up in WO-1's Board tableConfiguration). The Kanban, status updates, archive/delete/restore all use existing Job mutations — same editJob, archiveManyJob, etc. The only difference is which status PropertyValues load for the Kanban columns.

2. SharedJobListView Component

Implemented as: SharedListViews/JobList.tsx (named JobList to match the convention of other SharedListViews components like CompanyList, ContactList, etc.)

Props

interface IJobListProps {
    className?: string;
    subModel: string;               // "ProjectJob" | "WorkOrder"
    statusPropertyName: string;     // "job status" | "workOrderStatus"
    tableName: string;              // "Jobs" | "Work Orders"
    label: string;                  // "Jobs" | "Work Orders"
    createButtonText: string;       // "Create Job" | "Create Work Order"
    searchPlaceholder: string;
    detailsRoute: string;           // "/jobs" | "/work-orders"
    breadcrumbs: IBreadCrumbs;
    sessionFilterName: string;      // "jobs" | "workOrders"
    initialFilter?: Array<{ label: string; id: string; columnName: string; value: string }>;
}

Implementation notes

  • subModel is used for Board filtering (subModelForSave) and queryVariableDefaultFilters (jobClassification filter), NOT as the TableDataName
  • ListTableViews receives subModel="Job" (valid TableDataName) and subModelForSave={subModel} (for Board queries)
  • A jobClassification computed column on the Job TableFeed config handles the actual record filtering

What the component contains (extracted from pages/jobs/index.tsx)

  • ListTableViews with referenceModel="Job", subModel="Job", subModelForSave={subModel}
  • Status query using props.statusPropertyName for Kanban columns
  • editJob mutation for inline status updates
  • archiveManyJob / deleteManyJobs / restoreManyJob header actions
  • updateCardOrder mutation for Kanban drag
  • Edit handler opening DataModelForm flyout
  • Kanban card layout with detailsLink using props.detailsRoute
  • CSV export, views, filters

3. Kanban Board

Works automatically. The Kanban columns are loaded from PropertyValues filtered by statusPropertyName. Work Orders get: Draft, Submitted, Awaiting Signature, Signed, Approved, Invoiced, Void. Jobs continue to get their existing statuses.

Status updates use the same editJob mutation — both write to jobStatusId. Card drag reordering uses the same upsertManyBoardCards mutation.

4. Create Work Order Button

  • Opens DataModelForm flyout with formType = "CREATE"
  • Pre-selects a WorkOrder-classified jobType (auto-select if only one exists)
  • The subModel = "WorkOrder" loads the correct DAAs (dateOfWork, labor, material, equipment, signatures)
  • On save, navigates to /work-orders/[id]
