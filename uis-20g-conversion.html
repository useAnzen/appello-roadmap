WO → Invoice / Change Order Conversion

One-click conversion from an Approved Work Order to a Simple Invoice or a Change Order. Generates line items from the WO's labor, material, and equipment items with rates resolved at conversion time.

Estimate: 3-4 days Requires: WO-4 (Detail Page), WO-2 (Line Items), UIS-15B (Equipment Invoice pattern), LEM-1/2

// Creates a Simple JobBill for the job

3. WO → Change Order Conversion

Business Question for Corey / Chris

What is the intended use case for converting a WO to a Change Order? Possible scenarios:

  • Extra work documentation: A WO represents unplanned extra work. Converting to a CO creates a formal change order record that feeds into the job's contract value and financial summary.
  • Client approval flow: The WO is sent to the client as a CO request. Once approved/won, it updates the job's contract value.
  • Just for record keeping: Create a CO to capture the WO value in the estimate/change order system for reporting.

Recommendation: Implement as a basic conversion that creates a CO with line items. This is a stepping stone toward the flexible EstimateChangeOrderRecord that has been requested by multiple companies. The CO detail page will need minor updates to display line items without an account hierarchy.

Schema change: accountItemId optional

// EstimateChangeOrderRecordValue — make accountItemId nullable:
accountItemId    String?    @db.VarChar(21)    // was: String (required)
AccountItem      AccountItem? @relation(...)    // was: AccountItem (required)

When accountItemId is null, the line item is a generic description + value entry with no account hierarchy mapping. This is the case for all WO-generated CO line items.

Line item mapping

The numeric field on EstimateChangeOrderRecordValue represents hours of work. It applies to labor and equipment (which are time-based). Materials have no hour relationship — numeric is left at 0.

WO Line ItemECORV.descriptionestimate ($)numeric (hours)
Structured Labor "[TradeLevel] — [RateLabel]" rate × (duration/3600) × qtyOfWorkers (duration/3600) × qtyOfWorkers
Blank Labor description from line item rate × (duration/3600) × qtyOfWorkers (duration/3600) × qtyOfWorkers
Materials material name or description total 0 (no hour relationship)
Structured Equipment "[Equipment] — [RateLabel]" rate × calculated qty calculated qty (in hours equivalent where applicable)
Blank Equipment description unitRate × quantity 0 (manual qty — no direct hour relationship)
Equipment numeric: For structured equipment items billed per HOUR, the numeric value is the hours. For items billed per DAY/WEEK/MONTH, the numeric could be the hours equivalent (e.g., 8 hrs for a full day) or left at 0 since "1 day" isn't cleanly expressed as hours. Evaluate during implementation — if it causes confusion in the CO detail display, default to 0 for non-hourly equipment.

Mutation

convertWorkOrderToChangeOrder(jobId: ID!): EstimateChangeOrderRecord

// Creates a ChangeOrder EstimateChangeOrderRecord for the job
// Generates one EstimateChangeOrderRecordValue per WO line item
//   with accountItemId = null, description = line description
//   estimate = dollar value, numeric = hours/quantity
// CO bid = sum of all line item estimate values
// Returns the created CO for navigation

EstimateChangeOrderDetails update

The CO detail page (EstimateChangeOrderDetails) currently renders line items grouped by AccountItem in an account hierarchy. For WO-generated COs where accountItemId = null, it needs to fall back to a flat list of line items showing description + estimate + numeric — no hierarchy, just rows.

Context: This is a basic version of the "flexible EstimateChangeOrderRecord" that has been requested by multiple companies. Making accountItemId optional and showing line items without hierarchy is the first step. Full flexible estimating (where users can create free-form estimates without account items) is a future enhancement that builds on this change.

Confirmation modal

Both buttons show a confirmation modal before proceeding:

  • Summary of what will be created (line item count, total value)
  • "Confirm" / "Cancel" buttons
  • On confirm: mutation runs, user navigated to the created invoice or CO

Post-conversion state

  • After invoice generation: WO status transitions to "Invoiced". Generate Invoice button disabled. Convert to CO still available (can create CO after invoicing).
  • After CO conversion: WO status stays "Approved". A "Converted to CO #CO-042" badge appears. Convert to CO button disabled (one CO per WO). Generate Invoice still available.

6. Acceptance Criteria

  1. "Generate Invoice" button visible on WO detail header when status = Approved
  2. Confirmation modal shows line item count and total value before invoice generation
  3. Invoice created as Simple JobBill with one line item per WO line item
  4. Structured labor line items: rate from TradeLevelRate.billableValue, qty = hours × workers
  5. Structured equipment line items: rate from resolveEquipmentRate(), qty via calculateQuantity() (handles full-day vs prorated)
  6. Structured material line items: sell price from resolveMaterialSellPrice() (LEM-3 cascade), qty from line item
  7. Blank lines: rate/unitPrice + quantity from the line item directly
  8. After invoice generation: WO status → "Invoiced", button disabled
  9. User navigated to bill detail page after invoice creation
  10. "Convert to CO" button visible when status = Approved
  11. CO created as ChangeOrder with line items — accountItemId = null, description + estimate + numeric
  12. CO bid = sum of all line item estimate values
  13. accountItemId now nullable on EstimateChangeOrderRecordValue
  14. EstimateChangeOrderDetails renders flat line item list when accountItemId is null (no account hierarchy)
  15. After CO conversion: "Converted to CO #CO-042" badge appears, Convert to CO button disabled
  16. Both conversions can be done — invoice first, then CO (or CO first, then invoice)
