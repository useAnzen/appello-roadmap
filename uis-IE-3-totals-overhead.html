Totals & Overhead (Itemized)

Totals engine and overhead UI for creationType = Itemized estimates. Rolls up line cost / sell / tax / discount into EstimateChangeOrderTotals; applies estimate-level discount stack with proportional tax split; computes overhead, grand total, gross margin. The Itemized enum value itself lives in IE-1 (consolidating all schema work); this spec is totals + UI + the transactional upsert surface.

1. Business Case ▼

Bid and job value must reconcile with the sum of line prices plus overhead, mirroring how the Full worksheet uses COGS subtotals.

2. Scope ▼

Decision: creationType is immutable — no mode switching

Full / BidFirst worksheets and Itemized line items carry fundamentally different data shapes (see IE-1's creationType branching note). A Full worksheet's EstimateChangeOrderRecordValue rows cannot be meaningfully mapped onto ItemizedEstimateChangeOrderRecordLineItem, nor vice versa — there's no shared concept of “line” between GL-scoped totals and a resolved catalog row. Once an estimate is created as one type it stays that type for life; if a user wants to switch, they delete the estimate and create a new one with the desired creationType.

Enforcement is handled upstream in IE-1: EstimateChangeOrderRecord.creationType carries canEdit: false, and a gate validator rejects line-item writes when the parent ECR isn't Itemized. IE-3 therefore has no “switch mode” UX, no confirm dialog, no data-migration logic — it simply reads creationType and renders.

Decision: Contingency fields (resolved)

Part 1 — separate from subsidy. Subsidy and contingency are distinct concepts and stay distinct columns. subsidy reads as “external funding source reducing the price”; contingency reads as “reserved margin for unknowns.” Future reporting wants them separated; conflating them destroys audit history. We add contingencyPercent + contingencyDollarValue as new columns; subsidy is untouched.

Part 2 — both line-level and estimate-level. Contingency fields exist at both scopes:

  • Per line (on ItemizedEstimateChangeOrderRecordLineItem, added by IE-1) lets estimators pad riskier scope (rough-in, unknowns) without touching safer lines.
  • Per estimate (on EstimateChangeOrderRecord, added by IE-3) stacks a global pad on top for unknowns that aren't line-attributable.

Dollar-or-percent, not both. Each scope independently accepts either a percent or a dollar value, never both at once. Validation enforces this XOR per scope on both models. (Existing overhead* fields remain as-is; they predate this rule and allow both.)

The totals engine renders two distinct contingency rows in the rollup, always shown (zero when empty) so the reader can tell which scope contributed — same “provenance visible” pattern used for line vs. estimate discounts.

In Scope

  • IN EstimateChangeOrderTotals.tsx (Itemized branch): numbered rollup matching the IE-2 canvas — line sell before discounts, line discounts subtotal (sum of every per-line DiscountAssignment amount), line sell after line discounts, cost subtotal, estimate-level discount subtotal (sum of the estimate-level stack; always rendered on its own row, never merged with line-level discounts), taxable sell after all discounts, tax subtotal rolled up from lines (sum of per-line cached taxValue; this is a weighted average of per-line rates, not a document-level tax control), subtotal, overhead, contingency, grand total, gross margin %. Tax and discount rollup rows are always distinct and labeled so the reader can tell which level the money came from.
  • IN Reusable totals engine module (pure functions) consumed by the totals panel, PDF (IE-4), and invoice conversion (IE-5) so the math is written once
  • IN Overhead % edit UI on the left sidebar (live re-calc on change; persist with record save)
  • IN Estimate-level discount stack control in the totals panel — consumes SD-1's DiscountAssignment against referenceModel = "EstimateChangeOrderRecord"
  • IN Section / subsection rollups — numbered subtotals per node, computed at read time by walking the IE-1 tree (parentRecordId)
  • IN Transactional upsert surface: either extend upsertFullEstimateChangeOrderRecord or add dedicated upsertItemizedEstimateChangeOrderRecord that writes the ECR header + full line-item tree + tax M2M + discount assignments in one transaction
  • IN Estimate-level contingency fields on EstimateChangeOrderRecord — contingencyPercent + contingencyDollarValue, both nullable, XOR validation (one or the other, never both)
  • IN Totals engine sums per-line contingency (from IE-1's line-level contingencyPercent / contingencyDollarValue) into a distinct rollup row, separate from the estimate-level contingency row; both rows always rendered
  • IN Default markup % flow (“Set default markup” affordance next to the grid or totals panel) with product-chosen scope rule (only-unset / all / new-only)

  • OUT creationType immutability + creation-type gate — IE-1 (canEdit: false + line-item gate validator)
  • OUT Line-item model, tax M2M, line-item validators — IE-1
  • OUT Discount model and attach / detach / suggest resolvers — SD-1
  • OUT Line editor (cascades, sub-rows, section tree controls) — IE-2
  • OUT PDF layout — IE-4
  • OUT Invoice mutation — IE-5

3.2 Default markup flow

Clicking Set default markup % opens a modal with a single numeric input prefilled from ECR.estimateMarkup and two explicit action buttons. This replaces the earlier "pick one of three product rules" approach — both actions are always available, user chooses intent at click time:

  • Update default for future lines — writes the new value to ECR.estimateMarkup. Existing lines are untouched; newly added lines after this point will seed their markupPercent from the new value.
  • Apply default to all lines now — writes ECR.estimateMarkup and bulk-overwrites every existing priced line's markupPercent with the new value (Custom lines included; Section rows unaffected). The line's taxValue snapshot recomputes for every affected line because price changes.

The modal never only bulk-applies without also updating the default — keeping the two in lockstep means future lines don't silently revert to an older seed after a bulk-apply.

4. Data Model ▼

The Itemized value on EstimateChangeOrderRecordCreationType and the canEdit: false immutability on creationType both live in IE-1. Line-level contingencyPercent / contingencyDollarValue also live in IE-1's migration (they're columns on ItemizedEstimateChangeOrderRecordLineItem). IE-3 adds only the estimate-level contingency fields below.

Existing field, new meaning: EstimateChangeOrderRecord.estimateMarkup

The existing estimateMarkup column is semantically overloaded by creationType and we're intentionally reusing it rather than adding a parallel field:

  • Full / BidFirst: applied multiplier on the document total (existing behavior — unchanged).
  • Itemized: per-line seed for new markupPercent at insert time; never applied to Itemized totals directly (each line's own markupPercent drives sell).

Both flavors continue to seed from Instance.defaultEstimateMarkup / Instance.defaultChangeOrderMarkup at ECR creation (existing beforeCreate hook — no new logic needed). Discipline to keep the overload safe:

  • Schema comment on estimateMarkup in schema.prisma documenting both meanings
  • Top-of-file comment in itemizedTotalsEngine.ts noting that ECR.estimateMarkup is not consulted on the Itemized math path — it only governs new-line seeding (IE-2)
  • Same note at IE-4 (PDF) and IE-5 (invoice) integration points

On Itemized records, the existing estimateMarkup header input in EstimateChangeOrderDetails is hidden. The only edit surface is IE-3's “Set default markup %” modal (§3.3).

// Estimate-level contingency on the existing EstimateChangeOrderRecord model.
//   • Distinct from `subsidy` (external funding) — reserved margin for unknowns.
//   • XOR rule: at most one of (contingencyPercent, contingencyDollarValue)
//     may be non-null on a given ECR.
model EstimateChangeOrderRecord {
  // ... existing fields ...
  contingencyPercent      Decimal?  @db.Decimal(10, 2)
  contingencyDollarValue  Decimal?  @db.Decimal(10, 2)
}

// Line-level contingency lives in IE-1 (ItemizedEstimateChangeOrderRecordLineItem).
// Same two nullable decimals, same XOR rule. IE-3's totals engine sums them into
// the "Line contingency" rollup row; IE-1's migration owns the columns.

XOR validation fires on both models: when one of the two contingency fields is set, the other must be null. Mirrors the pattern in place for the Discount value resolution (percent vs dollar is a type distinction, not a dual value). Existing overhead* fields are exempt from this rule — they predate it and are allowed to coexist.

5. API Surface ▼

input UpsertItemizedEstimateChangeOrderRecordInput {
  estimateChangeOrderRecordId: ID!   # creationType is NOT passed — immutable (IE-1)

  # Header fields
  overheadMarkup: Float
  overheadDollarValue: Float
  estimateMarkup: Float
  markupDollarValue: Float
  contingencyPercent: Float
  contingencyDollarValue: Float
  bid: Float

  # Full tree write — sections + lines + tax set + line-level discount set.
  # parentRecordId / orderNo drive hierarchy; implementation walks depth-first
  # and issues creates/edits/deletes + connectTaxTypes + attach/detach discount
  # within one Prisma $transaction.
  lineItemTree: [UpsertItemizedEstimateChangeOrderRecordLineItemInput!]!

  # Estimate-level discount stack (SD-1 DiscountAssignment).
  estimateDiscountAssignments: [UpsertEstimateDiscountAssignmentInput!]
}

Two implementation options (plan will pick): (a) extend upsertFullEstimateChangeOrderRecord with the above fields when creationType = Itemized; (b) add a dedicated upsertItemizedEstimateChangeOrderRecord mutation. Option (b) reads cleaner given the shape is so different from the Full worksheet's EstimateChangeOrderRecordValue rollup.

Queries: IE-3's totals panel reads the tree via IE-1's itemizedEstimateChangeOrderRecordLineItems and estimate-level discounts via SD-1's discountsForEntity("EstimateChangeOrderRecord", id). No new read queries needed.

7. Acceptance Criteria ▼

  • Itemized totals panel renders a numbered rollup matching the IE-2 canvas: extended sell after line-level discounts → cost subtotal → estimate-level discount stack → taxable sell after estimate discounts → aggregated tax per TaxType → subtotal → overhead → line contingency (sum of per-line contingencyPercent / contingencyDollarValue) → estimate contingency → grand total → gross margin %. Line contingency and estimate contingency appear as two distinct rows, always rendered (shows $0.00 when empty).
  • Tax aggregation groups lines by TaxType; compound taxes apply to base + sum(non-compound) per the IE-1 / SimpleInvoice.tsx rule
  • Line discounts rollup (sum of per-line DiscountAssignment amounts) and estimate-level discount rollup (sum of the stack against EstimateChangeOrderRecord) appear as two distinct rows in the totals panel — never merged. Each row is always rendered (shows $0.00 when empty) so the reader can always tell which level the discount came from.
  • Tax rollup is always derived from lines: panel shows the sum of per-line cached taxValue with a weighted-average rate annotation. There is no estimate-level tax control; tax never originates outside the line grid.
  • Estimate-level discounts apply in orderNo order to the pre-tax sell; each line's taxable base shifts by its proportional share of the estimate-level discount; aggregated tax reflects the shifted bases
  • Section / subsection rollups computed at read time by walking IE-1's parentRecordId tree — no stored subtotals
  • Overhead % edit on the sidebar re-computes the panel live; persists on record Save
  • Transactional upsert writes header + tree + line tax sets + line discount assignments + estimate-level discount assignments in one Prisma $transaction; partial failure rolls everything back
  • Gross margin % matches the agreed formula (document in plan once chosen)
  • Totals engine is a pure-function module importable by PDF (IE-4) and invoice conversion (IE-5) without duplicating math
  • Estimate-level contingency fields round-trip through the upsert; XOR validation on EstimateChangeOrderRecord rejects any write where both contingencyPercent and contingencyDollarValue are non-null
  • Line-level contingency fields on ItemizedEstimateChangeOrderRecordLineItem (owned by IE-1) round-trip through the upsert tree; XOR validation applies per line
  • Engine test: a line with contingencyPercent = 10 and a price of $1,000 contributes $100 to the "Line contingency" rollup row
  • Engine test: an ECR with contingencyDollarValue = 500 contributes $500 to the "Estimate contingency" rollup row; with contingencyPercent = 3 instead, contributes 3% of the pre-contingency subtotal
  • No “switch creation type” UI, no confirm dialog, no data-migration logic exists in IE-3 — the path is gated by IE-1's canEdit: false + gate validator
  • EstimateChangeOrderDetails hides the existing estimateMarkup header input on Itemized records; no UI path exists to edit it outside the “Set default markup %” modal
  • “Set default markup %” modal prefills from ECR.estimateMarkup; two distinct actions are both visible (Update default for future lines, Apply default to all lines now); bulk-apply shows a secondary confirmation before overwriting
  • “Update default for future lines” writes ECR.estimateMarkup only; no existing lines are touched
  • “Apply default to all lines now” writes ECR.estimateMarkup and bulk-overwrites every priced line's markupPercent; each affected line's taxValue snapshot recomputes (IE-1 invariant); Section rows are unaffected
  • ECR.estimateMarkup is not consulted by itemizedTotalsEngine — verified by inspecting the module (explicit top-of-file comment + a unit test confirming changing estimateMarkup without touching any line's markupPercent does not shift the computed grand total on an Itemized record)
