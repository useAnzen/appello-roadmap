ItemizedEstimateChangeOrderRecordLineItem Schema & Model

Introduce ItemizedEstimateChangeOrderRecordLineItem as the persisted backing store for itemized estimate rows (only used when EstimateChangeOrderRecord.creationType = Itemized, added in IE-3). A polymorphic priceReferenceModel + priceReferenceId pair points at rate-level rows (TradeLevelRate, MaterialAttributePrice, EquipmentRateValue); custom rows omit both. Server-side helpers resolve default unit cost from LEM rate tables.

1. Business Case ▼

Itemized quoting requires rows that are not constrained to a single AccountItem per line. A dedicated model keeps the legacy GL worksheet intact while enabling catalog-backed pricing.

Structured rows must pull unit cost from rate tables so estimators are not re-keying prices. — Product direction (LEM alignment)

2. Scope ▼

In Scope

  • IN Prisma model ItemizedEstimateChangeOrderRecordLineItem + enums EstimateLineItemType (Labor, Material, Equipment, Custom) and PriceReferenceModel (TradeLevelRate, MaterialAttributePrice, EquipmentRateValue)
  • IN Extend existing EstimateChangeOrderRecordCreationType enum with Itemized (all DB updates consolidated in IE-1; IE-3 consumes the enum value)
  • IN Make EstimateChangeOrderRecord.creationType immutable after create via canEdit: false on the ECR model's modelFields — enforces the project-wide “never switch creation types” rule in code, not just convention
  • IN Creation-type gate: write-side validator rejects ItemizedEstimateChangeOrderRecordLineItem writes when the parent ECR's creationType is not Itemized
  • IN Polymorphic priceReferenceModel + priceReferenceId (nullable for Custom); optional userId permitted only on Labor rows (never required); required estimateChangeOrderRecordId
  • IN LineItemNodeType enum (Section, LineItem) + self-referential parentId for section / subsection nesting (tree depth cap 3); section rollups computed at read time, never stored
  • IN Fields: description, quantity, unit, unitCost, totalCost, markupPercent, markupDollarValue, price, orderNo, soft delete flags
  • IN Many-to-many TaxType relation (mirrors JobBillLineItem) with cached taxValue (write-time snapshot of price × effective tax rates, compound-aware)
  • IN Model class + BaseModel list/get/create/edit patterns
  • IN Service resolveItemizedLineItemUnitCost() branching on priceReferenceModel (delegates to TradeLevelRate / MaterialAttributePrice / EquipmentRateValue resolvers)

Out of Scope

  • OUT Frontend editor (IE-2)
  • OUT PDF and invoice conversion (IE-4, IE-5)

Note: Optional userId on labor rows aligns with the user-first labor rule for WO line items when the same picker is reused in IE-2.

Note on instanceId. Intentionally omitted — the app is single-tenant today. Add later only if multi-tenant requirements emerge.

Note on tax. Mirrors JobBillLineItem: many-to-many TaxType relation (supports stacked / compound taxes) with a cached taxValue. Tax base is the line's price (post-markup) — industry standard. taxValue is a write-time snapshot: recomputed on create/edit when price, markup fields, or the TaxType set changes, but not when a TaxTypeItem rate is edited later. Keeps estimates as point-in-time documents and lets IE-5 invoice conversion be a straight copy. excludeTax is intentionally omitted — it is a legacy field on JobBillLineItem with no active consumers.

Sections & subsections (tree structure)

IE-2's canvas introduces + Section and + Sub nested groupers; sections and priced lines appear as siblings at any depth, and whole subtrees reorder together. We model this as a single polymorphic table with a self-referential parentId and a nodeType discriminator (Section or LineItem).

  • Section nodes carry a title (via description) and a sibling orderNo; every priced/tax/reference field must be null. They are grouper-only.
  • LineItem nodes behave as before; parentId places them under a Section (or null = root).
  • parentId must reference a Section — priced lines never have priced-line children.
  • Section rollups (subtotals, row numbering) are computed at read time, never stored on Section rows. Keeps IE-3 as the single source of total math and avoids cache-invalidation headaches.

Business Question for Corey / Chris

Tree depth cap. We've scaffolded for max depth 3 from the EstimateChangeOrderRecord root — so Section → Subsection → LineItem is the deepest valid chain. The IE-2 canvas uses exactly three depth classes (depth-1, depth-2, depth-3), which reads as the intended cap, but please confirm this isn't accidentally too restrictive for any real estimate structure you've seen in the field.

Section-level markup / discount. Out of scope for v1 — today only lines and the whole estimate carry markup, per IE-2's canvas note. The polymorphic table shape deliberately leaves markupPercent and markupDollarValue on every row, so if sections ever need their own markup override the schema absorbs it with zero migration — only the validation rule would flip. Good to know if you'd like that in v2.

Design decision: polymorphic price reference

Why priceReferenceModel + priceReferenceId instead of hard FKs to TradeLevelRate / MaterialAttributePrice / EquipmentRateValue:

Reason for decision. Keeps the schema open to new price-lookup sources (subcontractor quotes, vendor PO prices, job-scoped rate overrides, etc.) without a follow-up migration each time. Matches the pattern already adopted by LEM-2's EquipmentRateValue (referenceModel + referenceId), so the estimating stack uses one shape end-to-end.

Tradeoffs accepted.

  1. No database-level foreign key on priceReferenceId — referential integrity is enforced in application code via the PriceReferenceModel enum and resolver validation on every write.
  2. No native Prisma include for the price-source join — GraphQL resolvers must branch on priceReferenceModel to hydrate a union type. LEM-2 is already paying this cost, so the pattern is already established.

Rejected alternative. Three nullable FKs (tradeLevelRateId, materialAttributePriceId, equipmentRateValueId): gains DB-level integrity for today's three sources, but every new price source becomes a schema change, and readers have to inspect multiple fields to discover which one is populated.

6. Acceptance Criteria ▼

  • Migration creates ItemizedEstimateChangeOrderRecordLineItem with indexes on estimateChangeOrderRecordId and on (priceReferenceModel, priceReferenceId)
  • CRUD respects BaseModel permissions scoped to estimate permissions
  • Create/edit mutations reject writes where priceReferenceModel is set without priceReferenceId (or vice versa), and reject non-null values for lineItemType = Custom
  • resolveItemizedLineItemUnitCost returns billable rate by loading TradeLevelRate by id and applying LEM-1 agreement context for Labor
  • Equipment path loads EquipmentRateValue by id and delegates to resolveEquipmentRate (LEM-2) for unit conversion
  • Material path reads MaterialAttributePrice.value by id (LEM-3)
  • GraphQL layer hydrates a union/polymorphic field for the price reference (branch on priceReferenceModel)
  • Create/edit accepts taxTypeIds: [String]; model connects/disconnects TaxType and writes taxValue as a snapshot of price × effective TaxTypeItem rate(s), stacked per isCompound ordering
  • taxValue is not recomputed when a TaxTypeItem rate is edited or a newer effective-date row is added — estimate stays pinned to the rate effective at save time
  • Section nodes reject any priced / tax / price-reference field value; LineItem nodes require lineItemType
  • parentId must reference a Section on the same EstimateChangeOrderRecord; tree depth from the ECR root is capped at 3
  • Itemized exists on EstimateChangeOrderRecordCreationType; writes to ItemizedEstimateChangeOrderRecordLineItem are rejected when parent ECR's creationType is Full or BidFirst
  • EstimateChangeOrderRecord.creationType is immutable after create — edit payloads carrying a changed value are stripped by the base model (canEdit: false); stored value remains
