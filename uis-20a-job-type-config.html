1. Architecture: Three Layers of Job Classification

Job classification uses three complementary layers, each serving a different purpose:

LayerWherePurposeExample
jobClassification
enum on PropertyValue		 PropertyValue (jobType records) Drives system behavior: which list, which route, which features enabled ProjectJob vs WorkOrder
jobTypeId
FK on Job		 Job → PropertyValue User-facing label within a classification: the specific type the user selects when creating a job "T&M Work Order", "Fireproofing WO", "Commercial Construction"
DocumentAttribute
subModel on assignment		 DocumentAttributeAssignment Drives per-type form fields: which custom fields appear on the form based on the job type Insulation WO gets R-Value field, Fireproofing WO gets UL Rating field
"Each template should have its own form — the fields for insulation work orders are nothing like the fields for fireproofing." — AB Insulation (Requirements Workshop)

How it flows

Admin configures in Settings → Job Types:
  "T&M Work Order"         → jobClassification: WorkOrder
  "Fireproofing WO"        → jobClassification: WorkOrder
  "Commercial Construction" → jobClassification: ProjectJob
  "Residential"            → jobClassification: ProjectJob

User creates a Job:
  Selects jobType = "T&M Work Order"
  → System reads jobType.jobClassification = WorkOrder
  → Job appears on Work Orders list (not Jobs list)
  → WO-specific features enabled (WO statuses, line items, signatures)
  → DocumentAttribute fields for "T&M Work Order" subModel render on the form

GraphQL resolver:
  Job.jobClassification → job.jobType?.jobClassification ?? "ProjectJob"

TableFeed / SQL View:
  JOIN PropertyValue pv ON pv.id = j.jobTypeId
  WHERE pv.jobClassification = 'WorkOrder'  -- for WO list feed
No field on Job model: jobClassification is derived from job.jobType.jobClassification at query time via a field-level resolver on Job and a computed column in the TableFeed SQL view. This avoids denormalization and means changing a job type's classification automatically re-routes all jobs of that type.

2. Scope

In Scope (WO-1)

  • IN JobClassification enum: ProjectJob, WorkOrder
  • IN jobClassification nullable field on PropertyValue (only relevant for jobType PropertyValues)
  • IN Field-level GraphQL resolver: Job.jobClassification derived from jobType.jobClassification
  • IN Computed column in JobListFeed SQL view for TableFeed filtering
  • IN Admin UI: classification dropdown on Job Type settings (map each type to a classification)
  • IN Work Orders list page — new route, filters by jobClassification = WorkOrder
  • IN Jobs list filters to jobClassification = ProjectJob (existing page, updated filter)
  • IN Work Order-specific statuses seeded as PropertyValues (Draft, Submitted, Awaiting Signature, Signed, Approved, Invoiced, Void)
  • IN Sidebar navigation: add "Work Orders" entry

Already Done

  • PR #6831 projectId optional on Job — Work Orders can exist without a project
  • PR #6831 jobTypeId on Job with PropertyValue relation — user-facing job type selector
  • Existing DocumentAttribute system with subModel on DocumentAttributeAssignment — per-type form customization infrastructure

Out of Scope (v1) — Designed For

  • FUTURE Per-type form customization UI — admin configuring which DocumentAttributes appear for which jobType. Infrastructure exists, UI deferred.
  • FUTURE Additional classifications beyond ProjectJob and WorkOrder (e.g., TASK, SERVICE_CALL). Enum is extensible.
  • FUTURE Classification-specific form rendering — different base form layouts per classification. For v1, the standard Job form handles both with WO-specific fields gated by classification.

3. Data Model

3.1 JobClassification Enum

enum JobClassification {
  ProjectJob
  WorkOrder
}

Extensible — future values (TASK, SERVICE_CALL, etc.) can be added without schema migration by using ALTER TYPE or by switching to a string field with validation.

3.2 jobClassification on PropertyValue

// Add to existing PropertyValue model:
jobClassification    JobClassification?    // nullable — only relevant for jobType PropertyValues

When propertyName = "jobType", this field determines how the system treats jobs of this type. When null or on non-jobType PropertyValues, it's ignored. Default behavior for unclassified types: ProjectJob.

3.3 No Changes to Job Model

The Job model is not modified. Classification is always derived from the relation:

Job.jobTypeId → PropertyValue.jobClassification

A field-level GraphQL resolver on Job exposes jobClassification as a computed field. The SQL view for TableFeed adds a computed column via the join.

3.4 Migration

4. List Routing & Feature Gating

How classification drives the UI

ProjectJobWorkOrder
List pageExisting Jobs list (/jobs)New Work Orders list (/work-orders)
Status setExisting job statuses (PropertyValue)WO statuses: Draft → Submitted → Awaiting Signature → Signed → Approved → Invoiced → Void
Project requiredTypically yes (existing behavior)No — projectId optional (PR #6831)
Line items (WO-2)Not applicableWorkOrderLaborItem, MaterialItem, EquipmentItem
Signatures (WO-3)Not applicableSignature capture on WO form
Conversion (WO-7)Not applicableWO → CO / WO → Invoice conversion

GraphQL Resolver

// In Job GraphQL type resolvers:
jobClassification: async (parent) => {
    if (!parent.jobTypeId) return "ProjectJob";
    const jobType = await prisma.propertyValue.findUnique({
        where: { id: parent.jobTypeId },
        select: { jobClassification: true }
    });
    return jobType?.jobClassification ?? "ProjectJob";
}

SQL View: Computed Column for TableFeed

-- Add to JobListFeed.sql (or create WorkOrderListFeed.sql):
SELECT
    j.*,
    COALESCE(pv_type.jobClassification, 'ProjectJob') as jobClassification
FROM Job j
LEFT JOIN PropertyValue pv_type ON pv_type.id = j.jobTypeId
WHERE ...

The existing Jobs list feed adds WHERE jobClassification = 'ProjectJob'. The new Work Orders list feed adds WHERE jobClassification = 'WorkOrder'. Both use the same base query pattern via TableFeed.

5. Admin UI: Job Type Classification Mapping

In the existing Settings → Job Types management interface, add a "Classification" dropdown per job type.

  • Dropdown options: "Project Job" (default), "Work Order"
  • Displayed as a column in the job types list or as a field in the job type edit form
  • Saves to PropertyValue.jobClassification
  • Job types with no classification set default to ProjectJob
Future extensibility: When new classifications are added (TASK, SERVICE_CALL, etc.), the dropdown options expand. Existing job types keep their classification. The admin just maps new types to the new classification. No code changes — just an enum extension and new PropertyValue seeds.

6. Work Order Statuses

Work Orders use a different status workflow than Project Jobs. Seeded as PropertyValue records with propertyName = "workOrderStatus".

StatusWorkflow StateMeaning
DraftNotStartedWO created, not yet submitted for review
SubmittedInProgressWO submitted by field worker, awaiting office review
Awaiting SignatureInProgressSent to client/GC for signature
SignedInProgressClient has signed, awaiting internal approval
ApprovedCompletedInternally approved, ready for invoicing
InvoicedCompletedInvoice generated from this WO
VoidCancelledWO cancelled/voided
Status set is determined by classification: When a Job's jobClassification = WorkOrder, the UI renders WO statuses. When ProjectJob, the existing job statuses render. The status dropdown queries PropertyValues filtered by the appropriate propertyName ("workOrderStatus" vs "jobStatus").

9. Acceptance Criteria

  1. JobClassification enum exists with ProjectJob and WorkOrder values
  2. jobClassification nullable field on PropertyValue — only relevant for jobType PropertyValues
  3. Admin can set classification per job type in Settings → Job Types via dropdown
  4. Job types without a classification default to ProjectJob
  5. Job.jobClassification GraphQL field resolves from jobType.jobClassification
  6. Work Orders list page at /work-orders shows only jobs where jobClassification = WorkOrder
  7. Jobs list at /jobs shows only jobs where jobClassification = ProjectJob (or null/unset)
  8. Sidebar shows "Work Orders" navigation entry
  9. "+ New Work Order" button creates a job pre-set with a WorkOrder job type
  10. WO statuses seeded: Draft, Submitted, Awaiting Signature, Signed, Approved, Invoiced, Void
  11. Work Order jobs use WO statuses; Project Jobs use existing job statuses
  12. Changing a job type's classification re-routes all jobs of that type (no data migration needed)
  13. Existing Jobs list and job functionality completely unaffected for unclassified types
