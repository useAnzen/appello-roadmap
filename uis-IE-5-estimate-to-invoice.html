Itemized Estimate → Simple Invoice

createInvoiceFromEstimate(estimateChangeOrderRecordId) creates a JobBill with template: Simple and one JobBillLineItem per EstimateLineItem. Both sides are linked in one transaction: JobBill.sourceEstimateChangeOrderRecordId and EstimateChangeOrderRecord.generatedJobBillId — so the estimate flips from Generate invoice to View invoice and a second invoice cannot be created. Won is resolved via instance.estimateChangeOrderWonStatusId; the button is hidden for users without JobBill create permission. Confirmation modal before create (same UX pattern as WO-7).

Estimate: 2–3 days Parent: UIS-IE

1. Business Case ▼

Eliminates re-keying from winning estimate into billing. Clement and similar residential workflows need estimate-to-invoice without a progress report.

2. Scope ▼

In Scope

  • IN Mutation + model method on JobBill or EstimateChangeOrderRecord
  • IN Button on estimate detail when type = Estimate, creationType = Itemized, statusId === instance.estimateChangeOrderWonStatusId, user has create JobBill permission, and no invoice has been generated yet
  • IN Confirmation modal: line count, grand total, job selection (default estimate job)
  • IN Persist the generated JobBill link back on the estimate (generatedJobBillId) so the button flips to a View invoice link on subsequent visits

Out of Scope

  • OUT Progress-based invoice generation (existing flow)
  • OUT Automatic tax jurisdiction resolution beyond existing job bill tax UX
  • OUT Generating multiple invoices from the same estimate (single-shot only; once generatedJobBillId is set the action is gone)

3. UI Concepts

Primary action on EstimateChangeOrderDetails (header or Billing area). Modal pattern mirrors WO-7 ConversionConfirmationModal — summary first, destructive/irreversible action clearly labeled.

Visibility rules (all must be true to render the primary button):

  • type = Estimate and creationType = Itemized
  • statusId === instance.estimateChangeOrderWonStatusId (the instance-configured Won status, not a hardcoded enum)
  • User has create JobBill permission
  • generatedJobBillId is null (no invoice has been generated yet)
  • At least one line item exists

When generatedJobBillId is set, the action is replaced with a View invoice link pointing to that JobBill — see 3.4.

3.3 Error state

Because the button is already gated on permission, Won status, itemized creationType, and absence of a prior invoice, the remaining failure surface is narrow — effectively GraphQL/network faults or a server-side race (e.g. another user just generated an invoice for the same estimate). On failure: keep the modal open, show an inline error in the modal footer with a Retry button, and log the server message for support. Do not clear the summary.

Close

Note: “insufficient billing permission” is not a reachable error here — the button is hidden for users without create JobBill permission. If the server ever returns that code it’s a bug, not a user-facing state.

3.4 After invoice generated — View invoice link

Once generatedJobBillId is set on the estimate, the primary action is replaced with a View invoice link that deep-links to the associated JobBill detail. This is the permanent state for that estimate — a second invoice cannot be generated (see Scope). The link shows the invoice number sourced from the linked JobBill.

Rendered in place of Generate invoice in every surface that exposes the action (header toolbar + Billing area).

4. Line Mapping ▼

Each leaf ItemizedEstimateChangeOrderRecordLineItem (nodeType = "LineItem") becomes one JobBillLineItem. Section / SubSection parent nodes are skipped — Simple bills are flat. Per-line tax and discount carry over as a verbatim snapshot (IE-1 designs taxValue as a point-in-time snapshot; SD-1 lists JobBill / JobBillLineItem in the DiscountAssignment allowlist specifically for this conversion).

Estimate-level discount stack

Top-level DiscountAssignment rows attached to the EstimateChangeOrderRecord are cloned onto the new JobBill with referenceModel = "JobBill" and the same discountId values (overrideValue copied verbatim) — preserving both the ordering and the exact stack the customer agreed to.

Contingency carry-over

Estimates hold contingency at two scopes (per-line + estimate-level, each XOR percent-or-dollar). The bill doesn't preserve that granularity — it preserves the value. At generation time, IE-3's totals engine resolves both scopes into dollar amounts; their sum is written to a new nullable JobBill.contingencyDollarValue column. Invoice rendering adds this to the bill grand total so the billed amount equals the quoted amount. When the source estimate had no contingency at either scope, the column is null.

5. API Surface ▼

Server enforces the same gates the UI uses — anything else is a bug:

  • Estimate statusId === instance.estimateChangeOrderWonStatusId
  • creationType = Itemized, type = Estimate
  • User has create JobBill permission
  • Estimate generatedJobBillId is null (prevents races on concurrent click)
  • At least one line item

On success the resolver writes generatedJobBillId on the estimate and sourceEstimateChangeOrderRecordId on the bill in the same transaction.

mutation CreateInvoiceFromEstimate($estimateChangeOrderRecordId: ID!) {
  createInvoiceFromEstimate(estimateChangeOrderRecordId: $estimateChangeOrderRecordId) {
    success
    jobBillId
    invoiceNumber
    message
  }
}

# Read shape used by EstimateChangeOrderDetails to decide Generate vs View
query EstimateInvoiceState($id: ID!) {
  estimateChangeOrderRecord(id: $id) {
    id
    statusId
    creationType
    generatedJobBillId
    generatedJobBill {
      id
      invoiceNumber
    }
  }
  instance {
    id
    estimateChangeOrderWonStatusId
  }
  currentUserPermissions {
    canCreateJobBill
  }
}

6. Change Surface ▼

  • MOD apps/api/prisma/schema.prisma — optional sourceEstimateChangeOrderRecordId on JobBill; optional generatedJobBillId on EstimateChangeOrderRecord (nullable 1:1 link to the generated bill)
  • MOD JobBill.ts model — createInvoiceFromEstimate (single transaction: create bill + lines, set both link columns, enforce all gates incl. Won-status check against instance.estimateChangeOrderWonStatusId and existing-invoice guard)
  • MOD JobBill.ts GraphQL resolver — mutation + expose sourceEstimateChangeOrderRecord relation
  • MOD EstimateChangeOrderRecord.ts GraphQL resolver — expose generatedJobBillId + generatedJobBill relation
  • NEW EstimateInvoiceConfirmModal.tsx (or reuse pattern from WO conversion modal)
  • MOD EstimateChangeOrderDetails.tsx — swap primary action between Generate invoice and View invoice based on generatedJobBillId; hide entirely when any other gate fails

7. Acceptance Criteria ▼

  • Button only visible when statusId === instance.estimateChangeOrderWonStatusId, creationType = Itemized, user has create JobBill permission, generatedJobBillId is null, and at least one line exists
  • Won status is resolved from the Instance — never hardcoded
  • Users without create JobBill permission never see the button, and the resolver rejects the mutation if somehow invoked — UI must not surface “insufficient billing permission” errors
  • Created bill total matches sum of line prices (+ taxes per existing Simple rules)
  • Navigates to bill detail after success
  • On success, both JobBill.sourceEstimateChangeOrderRecordId and EstimateChangeOrderRecord.generatedJobBillId are written in the same transaction
  • After generation, Generate invoice is replaced in every surface (header toolbar + Billing area) with a View invoice INV-XXXX deep link to the generated JobBill
  • A second click during a concurrent race (e.g. two tabs) is rejected by the server guard on generatedJobBillId and surfaces the generic inline retry error
