Sales Discount Model

Introduce a reusable Discount entity and a polymorphic DiscountAssignment join (mirroring PurchaseOrder / PurchaseOrderAssignment) so sell-side discounts can be authored once and attached to any of: Company, Job, EstimateChangeOrderRecord, ItemizedEstimateChangeOrderRecordLineItem, JobBill, and JobBillLineItem. Foundation for IE-2's per-line and estimate-level discount UX, and for company-standing customer-rate tracking.

Estimate: 3–4 days Lane: CRM & Sales Blocks: IE-1, IE-2, IE-3, IE-5

Phase 1 scope — ad-hoc only

For the initial release, every Discount will be created ad-hoc inline from the IE-2 per-line or estimate-level picker (“Enter ad-hoc” → % / $ toggle + value). No management UIs ship in this phase for Company-standing rates, Job-wide arrangements, or reusable discount catalogs.

The polymorphic DiscountAssignment (supporting Company, Job, EstimateChangeOrderRecord, ItemizedEstimateChangeOrderRecordLineItem, JobBill, JobBillLineItem) is scaffolded now so future phases can layer admin surfaces on top without another migration: a Company-standing discount editor, a Job-level override, bulk promo management, suggestedDiscountsForLine hitting real catalogs instead of always returning ad-hoc. Model and resolvers ship now; the UIs for anything other than ad-hoc come later.

1. Business Case ▼

Construction and trade customers routinely carry standing discount arrangements — contractor rates, repeat-customer percentages, volume tiers, negotiated-per-project overrides, seasonal promotions. Today there is nowhere to record these as first-class data, so estimators retype them into free-text fields on every estimate. Two problems follow:

  • No consistency (the “BigCo 7.5%” turns into “BigCo 7%” the next time because someone mistyped)
  • No reporting (“what did customer discounts cost us last quarter?” is unanswerable)

IE-2's canvas also introduces two stackable discount concepts on an itemized estimate: per-line % discounts and estimate-level % / Fixed $ discounts (applied to pre-tax sell, with tax re-split proportionally). Rather than bake these into ItemizedEstimateChangeOrderRecordLineItem as scalar fields, we introduce a shared Discount entity with a polymorphic attachment so the same record can attach to a company standing rate, a single estimate, a single line, a job bill, or a job-wide arrangement.

We have contractor rates that apply to every job we send BigCo — we shouldn't have to re-apply 7.5% on every line or every estimate. And some one-off trade promos apply only for a few weeks. — Customer interviews, construction estimators

Scope note vs MAT-9. MAT-9 covers buy-side supplier discounts (DistributorDiscount — what we pay vendors, with Global / Category / Product scopes). SD-1 is sell-side — what customers pay us. Different side of the transaction, different model.

2. Scope ▼

In Scope

  • IN Prisma model Discount + enum DiscountType (Percent, FixedAmount) with name, description, value, optional effectiveFrom / effectiveTo, and isAdHoc flag
  • IN Prisma model DiscountAssignment with polymorphic referenceModel + referenceId, discountId, and orderNo for stack ordering
  • IN Validator enforcing referenceModel is one of the supported enum values, and that referenceId resolves to a live row in that table
  • IN Resolver discountsForEntity(referenceModel, referenceId) returning active assignments + hydrated Discount
  • IN Resolver suggestedDiscountsForLine(lineItemId) — unions discounts attached to the line's Job AND the Job's Company, de-duped by discountId, filters out isAdHoc = true and expired-by-date rows
  • IN Mutations: create / edit / archive / delete for Discount; attach / detach for DiscountAssignment; a convenience createAdHocDiscountAndAttach that creates an isAdHoc = true record and assignment in one call
  • IN BaseAuditableModel subclasses for both models with standard lifecycle hooks
  • IN GraphQL type registration following CLAUDE.md "Registering New GraphQL Files"

Out of Scope

  • OUT UI for managing the Discount library (a settings page listing reusable discounts) — belongs in a later UX spec; SD-1 exposes the CRUD, UX spec consumes it
  • OUT The discount application math (how a discount folds into totals, proportional tax split, compound ordering with multiple discounts) — belongs in IE-3's totals engine
  • OUT Auto-cascade behavior — a Discount attached to a Company does not auto-apply when a new estimate is created. It appears as a pickable option per IE-2 UX; user chooses to attach.
  • OUT Buy-side / supplier discounts — owned by MAT-9

3. Data Model ▼

enum DiscountType {
  Percent       // value = percentage (e.g. 5.00 = 5%)
  FixedAmount   // value = currency amount (e.g. 100.00 = $100)
}

model Discount {
  id            String       @id @db.VarChar(21)
  isDeleted     Boolean      @default(false)
  isArchived    Boolean      @default(false)
  createdAt     DateTime     @default(now())
  updatedAt     DateTime     @default(now()) @updatedAt

  name          String       @db.VarChar(191)    // "Contractor rate", "Repeat customer", ...
  description   String?      @db.Text
  type          DiscountType
  value         Decimal      @db.Decimal(10, 2)

  // When true, hidden from the suggestion picker. Used for inline one-offs
  // that a user typed directly on a line ("5% here only"). Still tracked
  // and auditable; can be promoted to a reusable Discount later.
  isAdHoc       Boolean      @default(false)

  // Nullable = open-ended. Used to auto-retire seasonal promos or
  // pre-schedule a contract rate that kicks in on a future date.
  effectiveFrom DateTime?
  effectiveTo   DateTime?

  DiscountAssignments DiscountAssignment[]

  @@index([isAdHoc, isArchived, isDeleted])
}

model DiscountAssignment {
  id             String   @id @db.VarChar(21)
  isDeleted      Boolean  @default(false)
  isArchived     Boolean  @default(false)
  createdAt      DateTime @default(now())
  updatedAt      DateTime @default(now()) @updatedAt

  // Polymorphic pointer to the attached entity. Validated in app code —
  // no DB FK because referenceModel names many tables.
  referenceModel String   @db.VarChar(191)
  referenceId    String   @db.VarChar(21)

  // Stack order when multiple discounts attach to the same reference.
  orderNo        Int      @default(0)

  discountId     String   @db.VarChar(21)
  Discount       Discount @relation(fields: [discountId], references: [id])

  @@unique([referenceModel, referenceId, discountId])
  @@index([referenceModel, referenceId])
  @@index([discountId])
}

Supported referenceModel values (v1)

  • Company — a customer's standing discount (surfaces in IE-2 picker for any estimate to that customer)
  • Job — project-specific negotiated rate
  • EstimateChangeOrderRecord — document-level / estimate-wide discount (stacks via orderNo)
  • ItemizedEstimateChangeOrderRecordLineItem — per-line % discount
  • JobBill — invoice-level discount
  • JobBillLineItem — per-invoice-line discount

New reference types are additive — no migration, just one new string. A dedicated DISCOUNT_REFERENCE_MODELS const in apps/api/src/types gates the allowed values at the app layer.

Why polymorphic vs hard FKs

Mirrors PurchaseOrder / PurchaseOrderAssignment. A hard-FK-per-reference-type design would mean companyId, jobId, estimateChangeOrderRecordId, itemizedLineItemId, jobBillId, jobBillLineItemId columns all nullable, exactly one populated per row — every new reference type becomes a schema migration. Polymorphic trades the DB-level FK for one app-layer validator and absorbs new reference types trivially.

Ad-hoc vs named discounts

When a user types “5%” directly on a line in IE-2, that still becomes a Discount row (with isAdHoc = true) + a DiscountAssignment. This keeps one query path for “all discounts on line X” and lets reporting treat them uniformly. The UI filters isAdHoc out of the suggestion picker so the library stays clean, and the model exposes a promoteAdHocDiscount(id, name) mutation if a user later wants to save the ad-hoc as reusable.

4. Resolution & cascade behavior ▼

“Suggested discounts for a line”

IE-2's per-line picker shows a union of:

  1. Discounts assigned to the line's parent Job (via DiscountAssignment where referenceModel = "Job", referenceId = line.Estimate.jobId).
  2. Discounts assigned to that Job's Company (via DiscountAssignment where referenceModel = "Company", referenceId = line.Estimate.Job.companyId).

De-duped by discountId. Filtered to active assignments (isDeleted = false, isArchived = false) whose Discount is also active and whose effectiveFrom / effectiveTo window includes today (or is open). Filtered to isAdHoc = false.

Attachment is explicit. Suggestions are pickable, not auto-applied. A company-wide 7.5% shows up as an option on every line for estimates to that company; the estimator clicks to attach, and it becomes a DiscountAssignment scoped to the specific line.

Application math lives in IE-3

SD-1 stores discounts; it does not compute totals. IE-3's totals engine consumes DiscountAssignment rows for an estimate and its lines, applies them in the IE-2 canvas's ordering (line % → line sell; estimate-level on pre-tax; tax on discounted share), and produces subtotals. SD-1 exposes the raw data; it does not opine on how it's summed.

Expiry handling

Queries that resolve active discounts filter out rows whose effectiveTo < now or whose effectiveFrom > now. An existing DiscountAssignment on a past estimate still resolves its discount as of the date the estimate was authored — resolution is point-in-time by write date of the assignment for estimate pricing, never live. (Invoice-side recompute follows the same rule; estimates are snapshot documents.)

5. API Surface ▼

type Discount {
  id: ID!
  name: String!
  description: String
  type: DiscountType!
  value: Float!
  isAdHoc: Boolean!
  effectiveFrom: DateTime

7. Acceptance Criteria ▼

  • Migration creates Discount and DiscountAssignment with all indexes and the (referenceModel, referenceId, discountId) unique constraint
  • CRUD for Discount respects BaseAuditableModel permissions; soft-delete cascades disable assignments from the resolvers
  • attachDiscount rejects writes whose referenceModel is not in the approved allowlist
  • attachDiscount rejects writes whose referenceId does not resolve to a live row in the table named by referenceModel
  • Same Discount cannot attach twice to the same reference (unique constraint enforces)
  • discountsForEntity returns active, non-archived assignments whose parent Discount is active and within its effective window
  • suggestedDiscountsForLine unions Job-level and Company-level discounts for the line's job, de-dupes by discountId, excludes isAdHoc, filters by effective date
  • createAdHocDiscountAndAttach writes an isAdHoc = true discount and its assignment in one transaction
  • promoteAdHocDiscount flips isAdHoc = false and requires a non-empty name
  • reorderDiscountAssignments mutates orderNo atomically across all provided ids for a given reference
