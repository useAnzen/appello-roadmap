Signatures & Submission

Signature section using the existing react-signature-canvas package and the polymorphic Signature model from WO-3a. Multiple signatures per WO. Draft/Submit status transitions with validation. Completes the WO create/edit form.

Sequence: WO-3c of 3 (final) Estimate: 2-3 days Requires: WO-3a (Signature model + DAA position), WO-3b (line item editors)

1. Signature Section Component

Replaces the <p>Signatures</p> placeholder in customFormControl["wo-signatures"]. Uses the existing SignatureQuestionType component from @useanzen/ui which wraps react-signature-canvas and handles asset creation/upload.

Component interface

interface IWOSignatureSectionProps {
    jobId: string;
    signatures: IWOSignature[];
    setSignatures: (sigs: IWOSignature[]) => void;
    createAsset: (variables) => Promise;
    updateAsset: (variables) => Promise;
    useGetUploadUrlQuery: UseGetUploadUrlQuery;
}

interface IWOSignature {
    id?: string;           // set when loaded from DB
    assetId?: string;      // set after canvas save
    userId?: string;       // selected user or null for external
    signerName: string;
    signedAt?: Date;
    latitude?: number;
    longitude?: number;
    isNew: boolean;        // local state flag
}

Behavior

  • Multiple signatures: "+ Add Signature" button adds a new signature slot. Each slot has its own canvas, signer selector, and clear button.
  • Signer selector: User dropdown (CustomCombobox searching Users) defaults to the current user. Can be changed to any user in the system, or left as a text-only name for external signers (client representatives).
  • Signature canvas: Uses the existing react-signature-canvas package (already installed via SignatureQuestionType). The component handles drawing, clearing, and converting to an image asset.
  • GPS capture: On signature save, capture browser geolocation (latitude/longitude) via the Geolocation API. Stored on the Signature record.
  • Asset creation: When the user finishes signing, the canvas is converted to a PNG, uploaded as an Asset via createAsset + upload URL, and the assetId is stored on the local signature state.
  • On form save: For each signature in local state that has an assetId, call createSignature mutation (from WO-3a) with referenceModel = "Job", referenceId = jobId.
  • Loading existing: When editing a WO, query signatures(referenceModel: "Job", referenceId: jobId) and display existing signatures as read-only images with signer name and timestamp.

2. Draft / Submit Status Transitions

Action buttons

ButtonBehaviorValidation
CancelDiscards unsaved changes, closes formNone
Save DraftSaves Job + line items + signatures with status = "Draft"Only name required
SubmitValidates all required fields, saves with status = "Submitted"Full validation (see below)

Submit validation rules

  • name — required (always)
  • dateOfWork — required on Submit
  • At least one line item (labor, material, or equipment) — required on Submit
  • Structured labor lines: trade level must be selected
  • Blank lines: description required
  • Material lines: quantity and unitPrice required

Save flow

On Save Draft or Submit:
  1. Validate (minimal for Draft, full for Submit)
  2. Save/edit Job via DataModelForm (includes dateOfWork, status)
  3. upsertWorkOrderLineItems({ jobId, laborItems, materialItems, equipmentItems })
  4. For each new signature with assetId:
       createSignature({ referenceModel: "Job", referenceId: jobId, assetId, userId, ... })
  5. Navigate to WO detail page or close form
Transaction behavior: Steps 2-4 should succeed or fail together. If line item save fails, the Job save should ideally be rolled back. In practice, since the Appello backend doesn't support multi-model transactions natively, error handling should notify the user and allow retry. The Job save (step 2) establishes the jobId needed for steps 3-4.

5. Acceptance Criteria

  1. "+ Add Signature" button adds a new signature slot with canvas and user selector
  2. User selector defaults to current user, can be changed to any user via dropdown
  3. Signature canvas captures drawing via react-signature-canvas
  4. "Clear" button resets the canvas for that signature slot
  5. Remove (×) deletes a signature slot from local state
  6. GPS coordinates captured on signature save (browser Geolocation API)
  7. Completed signatures uploaded as Assets and linked via Signature model on form save
  8. Multiple signatures per Work Order supported
  9. Existing signatures display as read-only images with signer name, timestamp, and GPS on edit
  10. Save Draft: Only name required. Saves Job + any entered line items + any completed signatures. Status = "Draft".
  11. Submit: Full validation (name, dateOfWork, at least one line item). Status = "Submitted".
  12. Save flow: Job save → upsertWorkOrderLineItems → createSignature for each new signature
  13. Complete form renders all sections in order: Work Order Info, Labor, Materials, Equipment, Files, Signatures, Action Buttons
