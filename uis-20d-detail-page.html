1 · Business Case & Customer Evidence ▼

The detail page is where PMs review, approve, and take action on work orders. It reuses the existing DetailsPageWrapper + DetailsPageTabs pattern from the Job detail page, adding WO-specific tabs (Line Items, Signatures) and header actions (Convert to CO, Generate Invoice, Print PDF). The status badge with workflow transitions gives PMs a clear view of where each work order stands.

Customer Evidence

"When I open a work order, I need to see the total value, hours worked, and status immediately — without clicking into sub-pages. Then I need one-click actions to convert it to a change order or generate an invoice." — Tear Systems (PM Workflow Review)
"The line items should be viewable but not editable on the detail page. Editing happens through the flyout form — I don't want accidental changes on the review screen." — AB Insulation (UX Feedback)
📊
At-a-Glance Summary
Overview tab shows total value, labor hours, costs, status, and date in a 2×3 card grid.
⚡
One-Click Actions
Header actions for Convert to CO, Generate Invoice, and Print PDF — no navigation required.
🔄
Status Workflow
Status badge shows current state with available transitions — Draft → Submitted → Signed → Approved → Invoiced.
🏗️
Existing Patterns
Built on DetailsPageWrapper/Tabs — consistent with Job, Project, and Estimate detail pages.

2 · Scope ▼

In Scope

  • IN WO-specific tab ordering: Overview, Line Items, Signatures, Files, Settings
  • IN Header actions: Convert to Change Order, Generate Invoice, Print PDF
  • IN WO status badge with workflow transitions displayed in header
  • IN Overview tab with 2×3 summary card grid (Total Value, Labor Hours, Material Cost, Equipment Cost, Status, Date of Work)
  • IN Line Items tab with 3 collapsible sections (Labor, Materials, Equipment) — read-only tables
  • IN Signatures tab displaying captured signatures with signer names and timestamps
  • IN Edit button that opens WorkOrderForm flyout (from WO-3) in edit mode

Out of Scope

  • OUT Inline editing of line items on the detail page (editing happens via flyout)
  • OUT Audit trail / activity log tab (Phase 2)
  • OUT Custom tab ordering per user preference

4 · Data Model ▼

No new schema models are introduced. The detail page extends the existing Job detail page rendering, conditionally showing WO-specific content when isWorkOrder === true.

Conditional Rendering Logic

// In JobDetailsPage/index.tsx
const job = data?.job;

// Determine tab set based on job type
const tabs = job?.isWorkOrder
  ? ["Overview", "Line Items", "Signatures", "Files", "Settings"]
  : existingProjectJobTabs;

// Determine header actions based on job type
const headerActions = job?.isWorkOrder
  ? [
      { label: "Convert to CO", onClick: handleConvertToCO },
      { label: "Generate Invoice", onClick: handleGenerateInvoice },
      { label: "Print PDF", onClick: handlePrintPDF }
    ]
  : existingProjectJobActions;

5 · API Surface ▼

This slice consumes existing queries from WO-1 and WO-2. No new API endpoints are introduced.

PDF Generation

The Print PDF action extends the existing PDF generation endpoint to support a WO-specific template. This template includes the header info, all 3 line item tables, summary calculations, and captured signatures.

7 · Acceptance Criteria ▼

  • When navigating to a Job where isWorkOrder === true, the detail page renders WO-specific tabs (Overview, Line Items, Signatures, Files, Settings)
  • The header displays the WO number, name, status badge (with color coding), and company name
  • Header actions (Convert to CO, Generate Invoice, Print PDF) are visible and functional
  • The Overview tab shows a 2×3 summary card grid with correct values (Total Value, Labor Hours, Material Cost, Equipment Cost, Status, Date of Work)
  • The Line Items tab shows 3 collapsible sections with read-only tables that match the data from WO-2 queries
  • The Edit button opens the WorkOrderForm flyout (from WO-3) pre-populated with the existing work order data
  • Existing project-based Job detail pages are unaffected — they continue to render with the default tab set
