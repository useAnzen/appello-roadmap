1 · Business Case & Customer Evidence ▼

Mid-market contractors carry inventory across warehouses, staging areas, and job trailers. Without tracking, they over-order (wasted capital), under-order (delayed jobs), or make truck rolls just to check stock.

Customer Evidence

"Goal: Implement a true inventory management system to track items and costs in real-time." — A&K Insulation (Demo, Nov 2025)
"On the material request, isn’t that the time where the person says, this can be fulfilled from inventory, this needs a PO?" — Corey Shelson
"Mobile material ordering, field requests, inventory, warehouse" — Brunco Insulations (Demo, Mar 2025)

Value Delivered

📦
Stock Visibility
What’s on hand and where — across warehouses, staging areas, and job trailers.
📝
Movement Ledger
Append-only audit trail of every receive, checkout, transfer, and adjustment.
🔔
Min/Max Alerts
Reorder triggers when stock falls below minimum thresholds.
🛠
Job Checkout with Cost Coding
Materials checked out to jobs are automatically coded to the correct cost code for financial tracking.

2 · Scope ▼

In Scope

  • IN InventoryLocation, InventoryItem, InventoryMovement Prisma models
  • IN InventoryStockLevel SQL view (aggregates movements per location per material)
  • IN Receiving auto-creates inventory (from MAT-6 Supplier PO Receiving)
  • IN Job checkout with cost code assignment
  • IN Transfer between locations
  • IN Physical count adjustment
  • IN Min/max alert thresholds per item per location
  • IN Stock levels dashboard page
  • IN Location detail page
  • IN Movement history ledger
  • IN Mobile QR stock check (using UIS-15E QR pattern)

Out of Scope

  • OUT Barcode label printing
  • OUT Multi-unit-of-measure conversions
  • OUT Lot/serial number tracking
  • OUT Automated reorder PO generation
  • OUT Bin/shelf sub-locations

4 · Data Model ▼

InventoryLocation

model InventoryLocation {
  id          String    @id @db.VarChar(21)
  isDeleted   Boolean   @default(false)
  isArchived  Boolean   @default(false)
  createdAt   DateTime  @default(now())
  updatedAt   DateTime  @default(now()) @updatedAt
  name        String    @db.VarChar(255)
  typeId      String?   @db.VarChar(21)
  address     String?   @db.Text
  notes       String?   @db.Text
  Items       InventoryItem[]
  @@index([isDeleted])
}

InventoryItem

model InventoryItem {
  id                String    @id @db.VarChar(21)
  isDeleted         Boolean   @default(false)
  createdAt         DateTime  @default(now())
  updatedAt         DateTime  @default(now()) @updatedAt
  locationId        String    @db.VarChar(21)
  Location          InventoryLocation @relation(fields: [locationId], references: [id])
  materialId        String    @db.VarChar(21)
  Material          Material  @relation(fields: [materialId], references: [id])
  quantityOnHand    Decimal   @db.Decimal(10, 2)
  minQuantity       Decimal?  @db.Decimal(10, 2)
  maxQuantity       Decimal?  @db.Decimal(10, 2)
  unitCost          Decimal?  @db.Decimal(10, 2)   // weighted average cost at this location
  // Note: sell price for inventory valuation is resolved via LEM-3 cascade
  // (resolveMaterialSellPrice) at query time — not stored on InventoryItem
  @@unique([locationId, materialId])
  @@index([locationId, isDeleted])
  @@index([materialId, isDeleted])
}

InventoryMovement

model InventoryMovement {
  id            String    @id @db.VarChar(21)
  isDeleted     Boolean   @default(false)
  createdAt     DateTime  @default(now())
  updatedAt     DateTime  @default(now()) @updatedAt
  type          String    @db.VarChar(50)
  materialId    String    @db.VarChar(21)
  quantity      Decimal   @db.Decimal(10, 2)
  fromLocationId String?  @db.VarChar(21)
  toLocationId  String?   @db.VarChar(21)
  jobId         String?   @db.VarChar(21)
  costCodeId    String?   @db.VarChar(21)
  referenceId   String?   @db.VarChar(21)
  referenceType String?   @db.VarChar(50)
  performedById String    @db.VarChar(21)
  notes         String?   @db.Text
  @@index([materialId, isDeleted])
  @@index([fromLocationId, isDeleted])
  @@index([toLocationId, isDeleted])
}

Movement Type Constants

ConstantDescription
ReceiveInbound from supplier PO receiving (auto-created by MAT-6)
CheckoutOutbound to a job with cost code assignment
TransferMovement between two inventory locations
AdjustmentPhysical count correction (positive or negative)
ReturnInbound from a vendor credit return (MAT-8)

InventoryStockLevel SQL View

Aggregates InventoryMovement records per location per material to produce a real-time quantityOnHand. Receives sum to location; checkouts subtract from location; transfers subtract from source and add to destination; adjustments apply directly.

7 · Acceptance Criteria ▼

  • Receiving a supplier PO (MAT-6) auto-creates an InventoryMovement of type Receive and updates stock at the destination location
  • Checkout deducts quantity from the source location and creates a Checkout movement linked to a job and cost code
  • Transfer creates two linked movements: outbound from source and inbound to destination, both referencing each other
  • Physical count adjustment creates an Adjustment movement with signed quantity (positive or negative) and updates quantityOnHand
  • Min/max alerts trigger when quantityOnHand falls below minQuantity on any InventoryItem
  • The lowStockAlerts query returns all items below their minimum threshold with deficit amounts
  • Movement ledger is append-only — movements are never edited or hard-deleted, only soft-deleted
  • Stock levels dashboard shows all locations with item counts and below-min counts
  • Location detail page shows all items at that location with on-hand, min, max, and status
  • Checkout assigns cost code to the movement and is reflected in JobFinancialSummaryService
  • Mobile QR stock check allows scanning a location QR code to view stock levels (UIS-15E pattern)
  • Checkout quantity cannot exceed quantityOnHand at the source location (server-side validation)
  • Transfer quantity cannot exceed quantityOnHand at the source location (server-side validation)
  • Permission checks enforced: inventory.view, inventory.create, inventory.edit, inventory.delete
  • All movements record performedById from the authenticated user context
  • InventoryStockLevel SQL view correctly aggregates all movement types to produce accurate on-hand quantities
