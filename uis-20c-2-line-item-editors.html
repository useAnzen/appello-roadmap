Line Item Editors

Custom input components for Labor, Material, and Equipment line items that plug into the DataModelForm via customFormControl. Labour and Equipment follow the same dual-source pattern: structured entries via the existing timesheet infrastructure (TimeSheetRecord for labour, EquipmentTimeEntry for equipment) plus blank lines (WorkOrderLaborItem, WorkOrderEquipmentItem) for ClearStory-style flexible entry. Material supports structured (from catalog) and blank (manual) lines. Subtotals render within each editor.

Sequence: WO-3b of 3 Estimate: 7-9 days Requires: WO-3a (DAA positions), WO-2 (blank-line models), EQ-0 (EquipmentTimeEntry)

1. How Editors Plug Into the DataModel Form

WO-3a created 4 DAA positions (wo-labor-items, wo-material-items, wo-equipment-items, wo-signatures) with placeholder <p> tags in customFormControl. WO-3b replaces the first three with real editor components.

Integration pattern (from PR #6831)

const customFormControl = {
    "wo-labor-items": (formData, setFormData) =>
        <LaborItemEditor
            jobId={jobId}
            items={laborItems}
            setItems={setLaborItems}
        />,

    "wo-material-items": (formData, setFormData) =>
        <MaterialItemEditor
            jobId={jobId}
            items={materialItems}
            setItems={setMaterialItems}
        />,

    "wo-equipment-items": (formData, setFormData) =>
        <EquipmentItemEditor
            jobId={jobId}
            items={equipmentItems}
            setItems={setEquipmentItems}
        />,

    "wo-signatures": () => <p>Signatures</p>  // WO-3c
};

Key behavior rules

  • Labour & Equipment — two paths each: Structured entries are created via existing timesheet infrastructure (TimeSheetRecord for labour, EquipmentTimeEntry for equipment) and saved immediately. Blank lines are maintained in local state and saved on form submit via upsertWorkOrderLineItems.
  • Material — local state: Both structured and blank material items maintain local state. Nothing saves to the database until the parent form's Save Draft or Submit is clicked.
  • Subtotals within editors: Each editor renders its own subtotal at the bottom. Labour subtotal combines hours from TimeSheetRecords and blank lines. For equipment structured lines, subtotals are only shown if user has viewWageDetails permission. Material subtotals always shown (sell prices auto-populated from LEM-3 cascade). Cost shown alongside sell price if user has viewWageDetails.
  • On form save: The parent form collects blank labour, material, and blank equipment line item state and calls upsertWorkOrderLineItems (from WO-2). Structured labour and equipment entries are already persisted via their respective workflows.
  • Add dropdown: Labour uses "Log Time" / "Blank Line". Equipment uses "Log Equipment" / "Blank Line". Material uses "Structured Line" / "Blank Line".

2. Shared Component: TimeEntryTable

Reusable component: TimeEntryTable is extracted from the existing TimesheetModification / TimesheetRateLabelColumns pattern into a self-contained, reusable component. It is used in both the existing TimesheetAddEditForm and the new LaborItemEditor (WO-3b). The component handles user selection, per-row trade level, tabular rate-label hour entry, and optional cost code breakdown.

Component interface

interface ITimeEntryTableProps {
    jobId: string;
    viewOnly?: boolean;
    // Date-grouped data
    dateGroups: ITimeEntryDateGroup[];
    onDateGroupsChange: (groups: ITimeEntryDateGroup[]) => void;
    // Cost code support
    costCodes?: GqlCostCode[];
    showCostCodes?: boolean;
}

interface ITimeEntryDateGroup {
    workDate: string;              // YYYY-MM-DD
    rows: ITimeEntryRow[];
    isApproved?: boolean;          // all rows read-only if Approved
}

interface ITimeEntryRow {
    id?: string;
    userId: string;                // which user this entry is for
    tradeLevelId: string;          // selected trade level
    hours: Record<string, number>;  // rateLabelId → seconds
    costCodeRows?: ICostCodeRow[];
    // Derived per-row (loaded when user selected):
    // - EmployeeAgreement (from user)
    // - Active contract (from agreement + parent group's workDate)
    // - Available TradeLevels + RateLabels (from contract + wage region)
}

// Reuses existing ICostCodeRow from TimesheetAddEditForm/types.ts

Date-grouped layout

A WO may span multiple days, so entries are grouped by work date. Each date group is a collapsible section with its own table of rows.

  • Date group header: Date display (e.g., "Apr 7, 2026") with date picker, row count, total hours. IconTrash to remove the group (if not Approved). Approved groups render all rows read-only.
  • User column: User dropdown per row. Selecting a user loads their EmployeeAgreement. Combined with the group's work date, the active EmployeeAgreementContract is resolved to populate the Trade Level and Rate Label columns.
  • Trade level per row: Dropdown populated from the user's active contract for the group's work date. Users may work at different trade levels depending on the task — e.g., Journeyman on one entry, Foreman on another. Maps to the optional overrideTradeLevelId on TimeSheetDetailRecord (see WO-2 Step 0).
  • Rate label columns: Populated from the user's active contract, filtered by the job's wage region with fallback to default. One column per EmployeeAgreementRateLabel where isLineItem = false (e.g., Wages 1x, Wages 1.5x, Wages 2x). Uses the same FormControl type="numeric" step={0.25} input pattern.
  • Total column: Sum of all rate label hours for the row.
  • Cost code sub-rows: Optional. "Split by Cost Code" adds sub-rows beneath a row. Each sub-row has a cost code selector + the same rate label columns. Cost code hours must sum to the parent row's totals. Same auto-assignment logic as the existing timesheet form.
  • "+ Add Row" within a group adds a new user/trade-level row. "+ Add Date" adds a new date group.

Sharing with TimesheetAddEditForm

The WO time entry table and the existing timesheet form share the same underlying concepts (rate-label columns, trade-level selection, cost code breakdown), but there is a key structural difference: timesheets are per-user while the WO interface handles multiple users (foreman logging for crew). This means sharing may be possible for lower-level pieces (rate label input cells, cost code row/section, validation helpers) but the table-level component will likely differ between contexts.

  • Shareable: TimesheetRateLabelColumns (rate label input cells), TimesheetCostCodeRow / TimesheetCostCodeSection (cost code sub-rows), validation helpers (validateJobCostCodes, calculateCostCodeRowTotal, autoAssignHoursToSingleRow)
  • WO-specific: The table itself — has User + Trade Level columns per row, multi-user support, all rows save together with the WO form. The existing TimesheetModification component should remain unchanged.

The definitive mockup is in the LaborItemEditor section below (Section 3).

3. LaborItemEditor

Component interface

interface ILaborItemEditorProps {
    jobId: string;
    blankItems: IBlankLaborRow[];
    setBlankItems: (items: IBlankLaborRow[]) => void;
}

// Represents a WorkOrderLaborItem (blank/unstructured line)
interface IBlankLaborRow {
    id?: string;
    orderNo: number;
    description: string;
    qtyOfWorkers: number;
    submittedDuration: number;  // seconds
    rate: number;
}

Structured labour (inline time entry)

  • Embeds TimeEntryTable with the WO job pre-selected and date-grouped layout. Existing TimeSheetRecords for the job are loaded and grouped by workDate on mount. New date groups via "+ Add Date", new rows via "+ Add Row" within a group.
  • Date groups: Each group represents a work date. The WO may span multiple days — select a date, then add multiple users for that date. The group's work date determines which EmployeeAgreementContract is active for each user.
  • User per row: Selecting a user loads their EmployeeAgreement. Combined with the group's work date, the active contract is resolved to populate trade levels and rate labels.
  • Scheduling & permissions: If the user is scheduled to the WO job via JobSchedule, they enter time directly. If not scheduled, the unscheduled-job workflow applies based on permissions. If they lack permission, the "Log Time" option shows a permission message.
  • Approved rows read-only: Rows from Approved timesheets render with disabled inputs. Non-approved rows are fully editable.
  • Saves with form: No "Save Entry" button. All structured rows save together with the WO form via editFullTimeRecord mutation.
  • No rates shown: Hours only. Rates resolved from TradeLevelRate.billableValue at invoice generation (WO-7).

Blank line behavior

  • Description (free text, required), Workers qty, Hours (single input), Rate ($/hr, manual entry)
  • No agreement, no trade level, no rate label columns
  • Saves as a WorkOrderLaborItem with rate and description set — ClearStory-style flexible entry

4. MaterialItemEditor

Component interface

interface IMaterialItemEditorProps {
    jobId: string;
    items: IMaterialRow[];
    setItems: (items: IMaterialRow[]) => void;
}

interface IMaterialRow {
    type: "structured" | "blank";
    orderNo: number;
    materialId?: string;       // set for structured, null for blank
    description?: string;
    quantity?: number;
    unitPrice?: number;        // sell price (auto from LEM-3 cascade, editable)
    costPrice?: number;        // cost basis (read-only for margin visibility)
    total?: number;            // auto-calculated: qty * unitPrice
}

Structured line behavior

  • Material selector: CustomCombobox searching the Material catalog (Solr-indexed). Selecting a material auto-populates description (name) and UoM (from MaterialGroup).
  • Sell Price (unitPrice): Auto-populated from resolveMaterialSellPrice() (LEM-3 cascade: Job > Material > Group > cost pass-through). Editable — user can override the auto-populated value.
  • Cost (costPrice): Auto-populated from cost basis (DistributorPrice or MaterialAttributePrice). Shown read-only for margin visibility when user has viewWageDetails permission.
  • Total: Auto-calculated: quantity × unitPrice. Updated as the user types.

Blank line behavior

  • Description (free text, required), Qty, Unit Cost, Total (auto-calculated)
  • No material catalog lookup
  • Saves as WorkOrderMaterialItem with materialId = null

5. EquipmentItemEditor

Same dual-source pattern as LaborItemEditor: Structured equipment entries use EquipmentTimeEntry (from EQ-0). Blank lines use WorkOrderEquipmentItem (from WO-2). All rows are editable inputs and save together with the WO form — no separate "Save Entry" button. Includes cost code sub-rows via EquipmentTimeEntryCostCode.

Structured equipment (tabular entry)

  • Date-grouped layout mirroring the TimeEntryTable: Entries grouped by work date. Within each date group, all rows are editable inputs. Existing EquipmentTimeEntries loaded by jobId and grouped by workDate on mount. New date groups via "+ Add Date", new rows via "+ Add Row". Approved date groups render read-only.
  • Columns: Equipment (CustomCombobox, Solr-indexed), Rate Type (from EquipmentRateLabel for the equipment's category), Per (Hour/Day/Week/Month/Year), Duration (stored in seconds, auto-calculated for hour-meter equipment), optional Meter Start/End.
  • Cost code sub-rows: "Split by Cost Code" adds sub-rows beneath an equipment row. Each sub-row has a cost code selector + duration. Uses EquipmentTimeEntryCostCode from EQ-0. Same validation (cost code hours must sum to parent row duration) and auto-assignment logic as the labour TimeEntryTable.
  • Saves with form: No "Save Entry" button. All structured rows save together with the WO form via upsertManyEquipmentTimeEntries mutation.
  • No rates shown: Rates resolved from LEM-2 at invoice generation (WO-7).

Blank line behavior

  • Description (free text, required), Qty, Rate (CurrencyInput)
  • No equipment catalog lookup, no rate labels, no cost codes
  • Saves as WorkOrderEquipmentItem with unitRate and description set — for ad-hoc rentals or equipment not in the catalog

7. Acceptance Criteria

  1. TimeEntryTable: Date-grouped layout — entries grouped by work date, each group is a collapsible section with its own table
  2. TimeEntryTable: "+ Add Date" adds a new date group; "+ Add Row" adds a user/trade-level row within a group
  3. TimeEntryTable: User dropdown per row; selecting user + group work date resolves active EmployeeAgreementContract for trade levels and rate labels
  4. TimeEntryTable: Trade level dropdown per row; same user can appear multiple times with different trade levels (maps to optional overrideTradeLevelId on TimeSheetDetailRecord)
  5. TimeEntryTable: Approved date groups render all rows read-only
  6. TimeEntryTable: "Split by Cost Code" adds cost code sub-rows beneath a trade-level row; cost code selector excludes already-selected codes
  7. TimeEntryTable: Cost code hours validate against parent row totals before save (same logic as existing timesheet form)
  8. TimeEntryTable: Hours entered in 0.25 increments via numeric input, stored as seconds internally
  9. TimeEntryTable: Component is shared — used by both TimesheetModification (existing timesheet form) and LaborItemEditor (WO form)
  10. TimesheetModification: Refactored to wrap TimeEntryTable without regression — all existing timesheet functionality preserved
  11. Labor — Log Time: Inline TimeEntryTable with date-grouped editable inputs; existing TimeSheetRecords loaded and grouped by workDate on mount (Approved groups read-only); respects scheduling permissions
  12. Labor — Log Time: All structured rows save with the form via editFullTimeRecord — no separate "Save Entry" button
  13. Labor — Blank: Free-text description, manual hours + rate + workers qty; saved as WorkOrderLaborItem
  14. Labor — Subtotal: Combines hours from TimeSheetRecords and blank lines
  15. Material — Structured: Material catalog selector auto-populates name + UoM, sell price from LEM-3 cascade (editable), auto-calculated total
  16. Material — Blank: Free-text description, manual qty + unit cost + total
  17. Equipment — Structured: Date-grouped tabular layout with Equipment, Rate Type, Per, Duration. Existing entries loaded and grouped by workDate on mount. Approved date groups read-only. All save with the form.
  18. Equipment — Cost Codes: "Split by Cost Code" adds cost code sub-rows using EquipmentTimeEntryCostCode (EQ-0). Same validation + auto-assignment as labour.
  19. Equipment — Blank: Free-text description, manual qty + rate. Saved as WorkOrderEquipmentItem.
  20. Equipment — Subtotal: Combines hours/days from EquipmentTimeEntries and blank lines
  21. Labor uses "Log Time" / "Blank Line"; Equipment uses "Log Equipment" / "Blank Line"; Material uses "Structured Line" / "Blank Line"
  22. IconTrash delete button on blank lines and material/equipment rows deletes the item from local state
  23. Subtotals render at the bottom of each editor (hours for labor/equipment, dollars for materials)
  24. On form save: structured labour saved via editFullTimeRecord (creates TimeSheetRecord + detail records); structured equipment saved via upsertManyEquipmentTimeEntries; blank items saved via upsertWorkOrderLineItems (WO-2)
  25. Existing items loaded when editing: blank labour from WorkOrderLaborItem, structured labour from TimeSheetRecords, blank equipment from WorkOrderEquipmentItem, structured equipment from EquipmentTimeEntries, material from WorkOrderMaterialItem
  26. Structured entries and blank lines coexist in both labour and equipment editors
