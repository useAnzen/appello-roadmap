Work Order Line Item Tables

Material line items with rate resolution at invoice generation, plus unstructured blank lines for labour and equipment (ClearStory-style flexible entry) — the core data backbone of every T&M work order. Structured labour is handled by the existing JobSchedule + TimeSheetRecord / TimeSheetDetailRecord infrastructure. Structured equipment is handled by the existing EquipmentTimeSheetRecordValue infrastructure (EQ-0). The WO captures what was done (hours, quantities); pricing is applied downstream.

1 · Business Case & Customer Evidence ▼

Structured line items are what separates a first-class work order system from a form-based workaround. Without discrete labor, material, and equipment rows, there is no way to generate accurate invoices or feed into job P&L. The WO is a field document — it captures what was done (hours, equipment, quantities). Structured labour leverages the existing JobSchedule + TimeSheetRecord infrastructure so hours roll up into timesheets, the schedule, and payroll automatically. Structured equipment leverages the existing EquipmentTimeSheetRecordValue infrastructure (EQ-0) linked to TimeSheetDetailRecords. Unstructured (blank) labour lines use WorkOrderLaborItem and unstructured equipment lines use WorkOrderEquipmentItem for ClearStory-style flexible entry. Material has its own line item model with both structured and blank entries. Rates are resolved from the existing rate infrastructure (TradeLevelRate.billableValue via LEM-1 for timesheet-based labour, EquipmentRateValue for equipment via LEM-2) at invoice generation time (WO-7), not on the WO form itself.

2 · Scope ▼

Already Built

  • BUILT JobSchedule — assigns workers to jobs on specific dates. Existing desktop UI for scheduling and permissions for unscheduled job time logging.
  • BUILT TimeSheetRecord / TimeSheetDetailRecord — structured time entries with rate labels, trade levels, approval workflows, and payroll rollup.
  • BUILT Unscheduled job workflow — allows users with appropriate permissions to log time to jobs they aren't scheduled to.
  • BUILT EquipmentTimeSheetRecordValue (EQ-0) — structured equipment time entries linked to TimeSheetDetailRecords, with equipment rate labels, billing rules (per, operatorIncluded, isFullDayBilling), and rate resolution via LEM-2.

In Scope

  • IN 3 Prisma models: WorkOrderLaborItem (unstructured/blank lines only), WorkOrderMaterialItem, WorkOrderEquipmentItem (unstructured/blank lines only — structured equipment comes from EQ-0 EquipmentTimeSheetRecordValue)
  • IN Structured labour sourced from existing TimeSheetRecord / TimeSheetDetailRecord for the WO job — no new labour models for this path
  • IN Structured equipment sourced from existing EquipmentTimeSheetRecordValue linked to TimeSheetDetailRecords for the WO job (from EQ-0) — no new equipment models for this path
  • IN Upsert mutation for batch save of material, blank equipment, and blank labour line items
  • IN Recalculate totals mutation that queries both TimeSheetDetailRecords (structured labour) and WorkOrderLaborItems (blank lines) for the combined labour subtotal, and both EquipmentTimeSheetRecordValues (structured equipment) and WorkOrderEquipmentItems (blank lines) for the combined equipment subtotal

Out of Scope

  • OUT Drag-and-drop reordering (Phase 1 uses manual orderNo)
  • OUT Bulk import from external price sheets
  • OUT Custom column visibility per user
  • OUT New scheduling or time-logging UI — WO-3b embeds the existing log-time interface within the WO form

3 · UI Concepts ▼

All UI mockups for line item editors (labor, material, equipment) are in WO-3 (Create/Edit Form) since that spec owns the form interface. WO-2 defines the data models and API surface only.

Key UI patterns defined in WO-3:

  • Structured Labour: Embedded log-time interface within the WO form — creates TimeSheetRecord / TimeSheetDetailRecord entries for the WO job. Uses existing scheduling and unscheduled-job workflows based on user permissions.
  • Blank Labour: Free-text description + manual rate + hours + qty of workers. Stored as WorkOrderLaborItem records (ClearStory-style flexible entry).
  • Material: Structured lines (Material catalog lookup) and blank lines (free-text description + manual qty/cost). Same dropdown pattern.
  • Structured Equipment: Equipment time entries via EquipmentTimeSheetRecordValue (from EQ-0) linked to TimeSheetDetailRecords for the WO job — catalog equipment with rate labels, billing rules, and LEM-2 rate resolution.
  • Blank Equipment: Free-text description + manual qty + unit rate. Stored as WorkOrderEquipmentItem records (same pattern as blank labour).

Structured labour hours roll up into timesheets, scheduling, and payroll automatically. Blank labour lines and all material/equipment lines are WO-specific. No rates or dollar totals shown on the WO form — pricing applied at invoice generation (WO-7).

4 · Data Model ▼

Labour Architecture

Architecture decision: Work Order labour is split across two sources. Structured labour (trade-level, rate-label time entries) uses the existing JobSchedule + TimeSheetRecord / TimeSheetDetailRecord infrastructure — no new models needed. Hours roll up into timesheets, scheduling, and payroll automatically. Unstructured labour (blank lines) uses WorkOrderLaborItem for ClearStory-style flexible entry. The recalculateWorkOrderTotals mutation queries both sources.

TimeSheetDetailRecord — Optional Trade Level Override

⚠️ CALLOUT FOR CHRIS — PLEASE REVIEW ⚠️
Chris has been doing work around different wage rates per job/user. This change adds an optional tradeLevelId to TimeSheetDetailRecord to allow per-detail-record trade level selection when logging time from a Work Order. The existing derivation flow (UserTradeLevel → TimeSheetRecord.tradeLevelId) is NOT changed — this is additive only. Chris should review to ensure alignment with his recent work on per-job/user wage rates. See WO-3b LaborItemEditor mockup (Section 3) for how this is used in the UI.

Currently, trade level is derived from UserTradeLevel for the date and stored on TimeSheetRecord.tradeLevelId. In the WO context, a foreman may log time for multiple crew members, and some users work at different trade levels depending on the task. This optional override allows per-detail trade level selection without breaking existing behaviour.

WorkOrderLaborItem (Unstructured / Blank Lines Only)

model WorkOrderLaborItem {
  id                    String          @id @db.VarChar(21)
  isDeleted             Boolean         @default(false)
  isArchived            Boolean         @default(false)
  createdAt             DateTime        @default(now())
  updatedAt             DateTime        @default(now()) @updatedAt

  jobId                 String          @db.VarChar(21)
  Job                   Job             @relation(fields: [jobId], references: [id])

  rate                  Decimal?        @db.Decimal(10, 2)
  qtyOfWorkers          Int             @default(1)
  submittedDuration     Int             @default(0)    // seconds
  description           String?         @db.VarChar(500)
  orderNo               Int             @default(0)

  @@index([jobId, isDeleted])
}

This model is exclusively for unstructured (blank) labour lines — free-form entries where the user provides a description, rate, hours, and qty manually. No FK to TradeLevelRate. Structured labour lives in TimeSheetRecord / TimeSheetDetailRecord linked to the same Job.

WorkOrderMaterialItem (Revised)

Changes: Removed instanceId. Renamed unitCost to unitPrice (sell price, auto-populated from LEM-3 cascade, editable). Added costPrice (cost basis, read-only on WO). Structured material lines auto-populate both fields from resolveMaterialSellPrice() (LEM-3). Blank lines remain manual entry. 
model WorkOrderMaterialItem {
  id            String    @id @db.VarChar(21)
  isDeleted     Boolean   @default(false)
  isArchived    Boolean   @default(false)
  createdAt     DateTime  @default(now())
  updatedAt     DateTime  @default(now()) @updatedAt

  jobId         String    @db.VarChar(21)
  Job           Job       @relation(fields: [jobId], references: [id])

  materialId    String?   @db.VarChar(21)
  Material      Material? @relation(fields: [materialId], references: [id])

  description   String?   @db.VarChar(500)
  quantity      Decimal?  @db.Decimal(10, 2)
  unitPrice     Decimal?  @db.Decimal(10, 2)   // sell price (auto from LEM-3 cascade, editable)
  costPrice     Decimal?  @db.Decimal(10, 2)   // cost basis (from DistributorPrice or MaterialAttributePrice)
  total         Decimal?  @db.Decimal(10, 2)   // quantity * unitPrice
  orderNo       Int       @default(0)

  @@index([jobId, isDeleted])
}

Equipment Architecture

Architecture decision: Work Order equipment is split across two sources, same pattern as labour. Structured equipment (catalog equipment with rate labels and billing rules) uses the existing EquipmentTimeSheetRecordValue infrastructure from EQ-0, linked to TimeSheetDetailRecord entries for the WO job. Unstructured equipment (blank lines) uses WorkOrderEquipmentItem for ad-hoc entries with manual description, quantity, and rate. The recalculateWorkOrderTotals mutation queries both sources.

7 · Acceptance Criteria ▼

  • Unstructured (blank) labour items can be created with a description, manual rate, hours, and qty of workers
  • Material items can be created with a Material selection and the unit cost is auto-populated from the Material catalog
  • Blank equipment items can be created with a description, quantity, and manual unit rate
  • Row-level totals calculate correctly: Blank Labor = hours × rate × qtyOfWorkers; Material = Quantity × Unit Price; Blank Equipment = Quantity × Unit Rate
  • The upsertWorkOrderLineItems mutation saves blank labour, material, and equipment items in a single transaction
  • The recalculateWorkOrderTotals mutation queries both TimeSheetDetailRecords (structured labour) and WorkOrderLaborItems (blank lines) for the combined labour subtotal
  • The recalculateWorkOrderTotals mutation queries both EquipmentTimeSheetRecordValues (structured equipment) and WorkOrderEquipmentItems (blank lines) for the combined equipment subtotal
  • The recalculateWorkOrderTotals mutation updates the Job's laborSubtotal, materialSubtotal, equipmentSubtotal, and grandTotal
  • Deleting a line item (setting isDeleted: true) recalculates totals and removes the row from the UI
  • Structured labour hours logged via TimeSheetRecord for the WO job are included in the labour subtotal
  • Structured equipment hours logged via EquipmentTimeSheetRecordValue for the WO job are included in the equipment subtotal
