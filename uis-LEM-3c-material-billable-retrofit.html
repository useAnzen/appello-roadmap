Material Billable Retrofit

LEM-3a/3b were specced and approved after WO and IE work was already underway. Material columns in those features (and downstream reports) currently render raw cost — they need to route through resolveMaterialSellPrice() so customer-facing values reflect the configured markup. This spec catalogues every surface that needs the swap and the per-surface decision (show cost, show sell, show both, or show margin).

Estimate: 2-3 days (refine after audit) Sequence: After LEM-3a, WO-2, IE-2 are merged Requires: LEM-3a (resolver)
Why this exists. The original WO-2 / IE-1 / IE-2 specs were authored before LEM-3 was on the roadmap. Their line-item structures stored a single price (unitCost / unitPrice) that conflated cost and sell. Once LEM-3a’s resolveMaterialSellPrice() exists, the WO and IE line-item code, plus every report that consumes those rows, has to be retrofitted to use the resolved sell price for billable surfaces and the raw cost only for cost surfaces. Without this work, the markup configured via LEM-3a/3b will be silently ignored everywhere except the Material detail page.

1. Open Questions for Chris/Corey (block sign-off)

  1. Storage vs. compute. When a WO line item is saved with a structured material reference, do we (a) store both costPrice and unitPrice on the row at save time (snapshot), or (b) store only the cost and resolve sell on-read? Snapshot survives later markup changes; on-read always reflects current config. Recommendation: snapshot at conversion-time (WO → Invoice) and at IE submission, with a refresh-from-cascade button while editing.
  2. Column conventions on customer-facing PDFs. Does the customer-facing WO/IE PDF show only the sell column, or sell + a discreet “list” if a fixed override was used? Current LEM-3b mockup shows margin internally only.
  3. LEM Sheet. The LEM combined report needs both materialCost and materialBillableValue. MAT-2a originally owned the billable column; MAT-2a is now deferred. Does LEM-3c absorb the MAT-2a-scope LEM Sheet update for materials only, or does the LEM Sheet ship without a billable material column in 1.33?
  4. Backfill for already-saved rows. Any WO/IE rows created before 3a/3c lands have unitPrice equal to unitCost (or manual entry). Do we backfill via migration using the cascade, leave them as-is, or fire a one-time recompute on next edit?
  5. Override editability per row. When a structured material auto-populates unitPrice, can the user edit it inline on the line? If yes, which override field on the row records that — or is it just a free-form override that doesn’t persist back to Material.sellPriceOverride?

2. Audit Surfaces

First-pass list of every surface that touches a material price. Each gets a per-surface decision in §4. This list needs validation against the WO-2 / IE-2 / WO-7 PRs that landed during 1.33 build.

WorkOrderMaterialItem (WO-2) Schema: unitCost, unitPrice, quantity. On material select, unitPrice auto-populates from resolveMaterialSellPrice(materialId, jobId); unitCost from bestAvailableCost(materialId). Snapshot vs. on-read: TBD per Q1.
WO Line Item Editor (WO-3b) DataModel custom input for material lines. Uses the resolver on row construction. Inline edit story: TBD per Q5.
WO Detail Page line-item display (WO-4) Shows materials with their saved unitPrice. No retrofit if storage is snapshot; needs a re-resolve hook if on-read.
WO → Invoice / CO conversion (WO-7) Conversion has to call resolveMaterialSellPrice() for any structured line whose unitPrice wasn’t snapshotted. Blank/manual lines pass through unitPrice as authored.
ItemizedEstimate line items (IE-1, IE-2) Per-line markup & discount in IE compose with the LEM cascade. Open question: does the IE row use the cascaded sell as the base the IE markup is applied on top of, or does the IE markup replace the cascade entirely? Spec call needed.
IE PDF (IE-4) Customer-facing column visibility per Q2. Must not leak cost.
IE → Invoice (IE-5) Same snapshot question as WO-7.
LEM Sheet / LEMReport (was MAT-2a; MAT-2a now deferred) Decision per Q3. If retrofitted here: add materialBillableValue column alongside existing materialCost.
Job Financial Summary / P&L Cost-side already correct. Billable-side may aggregate from invoice journal entries; if it pulls from material rows directly, route through resolver.
JobBillLineItem (existing invoice lines) New invoices generated by WO-7 / IE-5 inherit the resolved value. Existing invoices are unchanged (do not retro-write).
MaterialOrder / MaterialOrderRecord Likely cost-side only. Confirm none of these views surface a sell price.

5. Acceptance Criteria

  • Every material-bearing surface listed in §2 has an explicit decision in §4 (cost / sell / both / margin) and an implementation that matches the decision
  • Selecting a structured material on a WO line item auto-populates unitPrice from resolveMaterialSellPrice(materialId, jobId) and unitCost from bestAvailableCost(materialId)
  • WO → Invoice conversion uses the resolved sell price for structured material lines; blank lines pass through manual unitPrice
  • IE line items reflect the cascade per the Q5 decision (composition rule documented and tested)
  • Customer-facing PDFs (WO / IE) do not expose cost or margin columns
  • Backfill / no-backfill decision is implemented per Q4 and explicitly noted in the migration / release notes
  • Tests exercise: group-default cascade, material override, job override, fixed sell, no-markup pass-through, and the WO/IE snapshot vs. on-read behavior
