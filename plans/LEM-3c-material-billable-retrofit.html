LEM-3c: Material Billable Retrofit — Plan

Self-contained execution plan. Cannot be built until the spec’s §1 open questions are answered — this plan assumes those decisions and lists what changes when.

Status: Review Required. The audit pass (Step 1 below) is the work that produces the actual code-change plan. Until Chris/Corey sign off on §1 of the spec and the audit lands, every “estimate” below is a placeholder.

Overview & Outcome

After this lands:

  • Every material-bearing surface (WO line items, IE line items, conversions, customer PDFs, LEM Sheet pending Q3) routes the displayed/billed value through resolveMaterialSellPrice().
  • Cost-side display continues to use bestAvailableCost() / MaterialAttributePrice as before — we do not collapse cost into sell.
  • WO → Invoice and IE → Invoice conversions snapshot the resolved sell price onto the new invoice/JobBill row at conversion time; later changes to material markup do not retroactively change posted invoices.
  • The retrofit does not break any existing invoice or finalized estimate.

Step 3 — API / service code edits

Goal
Route the right value through every API surface that emits a material price.
Files
WO-7 conversion service, IE-5 conversion service, possibly JobBillLineItem.ts, possibly JobFinancialSummaryService.ts.

Pattern for each:

// Before
const unitPrice = workOrderMaterialItem.unitPrice ?? workOrderMaterialItem.unitCost;

// After (snapshot fallback model)
let unitPrice = workOrderMaterialItem.unitPrice;
if (unitPrice == null && workOrderMaterialItem.materialId) {
    const r = await resolveMaterialSellPrice(
        workOrderMaterialItem.materialId,
        workOrderMaterialItem.workOrder.jobId
    );
    unitPrice = r?.sellPrice ?? workOrderMaterialItem.unitCost;
}

Audit each call site against the per-surface table from Step 1 — some surfaces want the on-read resolution every time (LEM Sheet), others want the snapshot (already-converted invoice).

Step 4 — Frontend edits

Goal
Auto-populate sell price on material select; honour the column-visibility decisions on customer-facing PDFs.

For the WO line item editor (and equivalent for IE):

  • On material select, fire resolveMaterialSellPrice(materialId: $materialId, jobId: $jobId) and populate unitPrice; populate unitCost from the costPrice in the same response (or via material.costBasis).
  • If Q5 allows inline override: keep the field editable; record any user edit on the row, do not write back to Material.sellPriceOverride.
  • If Q5 disallows inline override: render the field read-only with a tooltip pointing to the Material detail page’s Pricing card.

For PDFs (WO and IE):

  • Per Q2: hide cost / margin columns from customer-facing layouts; keep them on internal previews.

Step 5 — Backfill & release

Goal
Resolve in-flight WO/IE rows per Q4.

Three options — pick during audit:

  1. Backfill via migration. One-time UPDATE that sets unitPrice on existing structured rows by calling resolveMaterialSellPrice(). Requires raw SQL re-implementation of the cascade or a script that walks rows. Safe but invasive.
  2. Recompute on next edit. Leave existing rows alone; the next time someone opens and saves a WO/IE, the editor populates from the cascade. Cheapest, but stale rows remain stale until touched.
  3. Leave as-is. Existing rows authored before LEM-3 ship reflect the prices set at authoring time. Document this in release notes. Lowest risk for already-customer-facing documents.

Recommendation pre-audit: option 3 for already-finalized invoices/estimates, option 2 for drafts. Confirm with Corey.
