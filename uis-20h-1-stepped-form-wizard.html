Stepped Form Wizard

Adds a steppedForm mode to DataModelForm that renders each DocumentSection as a wizard step with Previous/Next navigation, progress indicators, and a final Summary step showing all entered data in read-only mode with per-section edit pencils. Reuses all existing DAA infrastructure — no custom step components needed.

Estimate: 3-4 days Requires: WO-3a (DAA sections), WO-3b (line item editors), WO-3c (signature section)

1. Approach

The DataModelForm already renders fields grouped by DocumentSection. In standard mode, all sections appear vertically in one scrollable form. The stepped form mode changes the rendering to show one section at a time, with navigation between them and a generated Summary step at the end.

Why not custom step components?

The WorkOrder subModel already has its DocumentSections configured (from WO-3a):

  1. Work Order Info — dateOfWork, description, job type, project, etc.
  2. Labor — customFormControl["wo-labor-items"]
  3. Materials — customFormControl["wo-material-items"]
  4. Equipment — customFormControl["wo-equipment-items"]
  5. Signatures — customFormControl["wo-signatures"]

By adding steppedForm to DataModelForm, we get a wizard for free from the existing section configuration. This approach also makes the stepped mode available to any future DataModelForm consumer, not just work orders.

Generic, not WO-specific. The steppedForm prop is a general enhancement to DataModelForm. It works with any referenceModel that has DocumentSections. Work Orders are just the first consumer. This means other forms (Estimates, Inspections) could opt into wizard mode in the future with zero additional work.

2. DataModelForm steppedForm Prop

New prop

interface IDataModelFormProps {
    // ... existing props from PR #6831
    steppedForm?: boolean;   // default false — renders all sections as wizard steps
}

Behavior when steppedForm = true

  • Section-as-step: Each DocumentSection (sorted by orderNo) becomes a numbered step. Only the active step's fields are visible.
  • Auto-generated Summary step: An additional final step is appended after all DocumentSections. This step shows a read-only recap of all entered data grouped by section.
  • Progress indicator: A step indicator (numbered dots connected by lines) shows at the top, with completed/active/pending states.
  • Navigation: Full-width "Previous Step" and "Next Step" buttons at the bottom of each step. First step hides "Previous". Summary step shows "Previous Step" only.
  • Header: Standard flyout header with form title. No custom EXIT button — the flyout's existing Cancel button in the footer handles dismissal.
  • Step validation: "Next Step" validates the current section's required fields before advancing. Invalid fields show inline errors and the step does not advance.
  • State persistence: All field values are held in the existing DataModelForm state across steps. Navigation between steps does not reset values.

What does NOT change

  • The underlying form state management, mutations, and save logic from PR #6831
  • customFormControl rendering — custom inputs still render within their section step
  • DAA field rendering — same components, same validation, just shown one section at a time
  • The form when steppedForm is false/undefined — completely unchanged

3. Step Indicator Component

A horizontal progress indicator showing all steps as numbered dots connected by lines. Sits below the mobile header, above the step title.

4. Step Rendering Logic

Pseudocode

// Inside DataModelForm when steppedForm = true:
const sections = documentSections.sort((a, b) => a.orderNo - b.orderNo);
const totalSteps = sections.length + 1; // +1 for Summary
const [activeStep, setActiveStep] = useState(0);

const isLastSection = activeStep === sections.length - 1;
const isSummary = activeStep === sections.length;

// Render:
if (isSummary) {
    return <SteppedFormSummary sections={sections} formValues={formState} onEdit={goToStep} onSubmit={handleSubmit} />;
}

const currentSection = sections[activeStep];
// Render only the DAAs belonging to currentSection
// (existing DataModelForm already groups DAAs by section — just filter to active)
return (
    <>
        <StepIndicator steps={totalSteps} activeStep={activeStep} onStepTap={goToStep} />
        <SectionTitle>{currentSection.name}</SectionTitle>
        {renderDAAsForSection(currentSection)}
        <StepNavigation
            onPrevious={activeStep > 0 ? () => setActiveStep(activeStep - 1) : undefined}
            onNext={() => validateAndAdvance()}
        />
    </>
);

Section field rendering

Each section renders exactly as it does in standard mode — DAA fields with their labels, validation, and customFormControl entries. The only difference is that non-active sections are hidden (not unmounted, to preserve state).

Do not unmount inactive steps. Unmounting would destroy local state in custom form controls (e.g., the labor item editor's draft rows). Use CSS display: none or conditional rendering that preserves the component tree. The form state itself lives in DataModelForm's state, but custom controls may have their own local state for UI interactions.

5. Summary Step

The auto-generated final step. Shows all entered data in read-only mode, grouped by DocumentSection. Each section group has an edit pencil icon that navigates back to that step for editing.

Layout per section

  • Section header: Section name (uppercase, small font) with edit pencil on the right
  • Standard DAA fields: Rendered as label/value pairs (read-only text, not inputs)
  • Custom form controls: Each custom control should implement a readOnlyRenderer or the summary falls back to showing the raw field value. For WO line items, the read-only view shows a compact list of items with quantities.

Actions on Summary step

  • "Submit Work Order" button — Creates the Job record, saves all line items and signatures, and sets status to Submitted (same as WO-3c submit logic). Single atomic save.
  • "Previous Step" — Goes back to the last section

7. Navigation & Validation

Step navigation rules

ActionBehavior
Next StepValidates current section's required fields. If valid, advances activeStep and scrolls to top. If invalid, shows inline errors on failing fields.
Previous StepMoves back without validation. Preserves all field state.
Tap completed dotJumps directly to that step without validation. Allows quick edits.
Edit pencil (Summary)Navigates to the corresponding section step. On return (via Next through remaining steps), lands back at Summary.
Cancel (flyout footer)Standard flyout cancel behavior. If form is dirty: discard confirmation. If clean: closes flyout.

Section-level validation

Validation only runs on "Next Step" and "Submit". Each section validates only its own DAAs. Custom form controls that need validation should implement it in their existing save/validate hooks — the stepped form calls the same validation the standard form uses.

No draft persistence in v1. The stepped form holds all state in memory. Exiting discards unsaved work. Draft save/resume functionality is deferred to the native app rebuild which will include offline support and local persistence.

8. Flyout Convention

The stepped form renders inside the standard mobile flyout. No custom header — uses the existing flyout header with the form title and the existing flyout footer Cancel button for dismissal.

Header title

For new records: "New [label]" (e.g., "New T&M Tag"). For editing: the record's display name (e.g., "WO-0042"). This uses the existing DataModelForm label or record name.

9. Desktop vs. Mobile Rendering

The steppedForm prop works on both desktop and mobile, but the rendering adapts:

AspectMobileDesktop
ContainerFull-screen view (not flyout)Standard flyout panel
HeaderEXIT / title / SAVE barStandard flyout header with close X
Step indicatorCompact numbered dotsSame dots, wider spacing
Navigation buttonsFull-width stackedRight-aligned inline
Summary sectionsFull-width cardsSame cards, max-width contained
V1 scope: Desktop stepped form rendering is a nice-to-have. The primary consumer is the mobile WO creation flow. If desktop stepped form adds significant complexity, it can ship as standard (non-stepped) form on desktop and stepped only on mobile. The prop can be conditionally set based on viewport or a useIsMobile() hook.

10. Work Order Consumer

The Work Orders create flow on mobile opens DataModelForm with:

<DataModelForm
    formType="CREATE"
    referenceModel="Job"
    subModel="WorkOrder"
    steppedForm={isMobile}
    customFormControl={{
        "wo-labor-items": (formData, setFormData, element, error) =>
            <LaborItemEditor formData={formData} setFormData={setFormData} element={element} error={error} />,
        "wo-material-items": (formData, setFormData, element, error) =>
            <MaterialItemEditor ... />,
        "wo-equipment-items": (formData, setFormData, element, error) =>
            <EquipmentItemEditor ... />,
        "wo-signatures": (formData, setFormData, element, error) =>
            <WOSignatureSection ... />,
    }}
    onSaved={(id) => router.push(`/work-orders/${id}`)}
/>

No separate summary renderers needed. The Summary step renders the same form sections in disabled mode — custom controls receive a no-op setFormData and display their current state as read-only.

On desktop, steppedForm is false, so the same form renders in the standard all-sections-visible flyout mode established in WO-3a/b/c.

Self-signing in stepped mode

The Signatures step (from WO-3c) includes the signature canvas and signer name input. In stepped mode, the user signs themselves as part of the wizard before reaching the Summary step. The signature is saved when they tap "Next Step" from the Signatures step. This is the "field worker signs their own work order" flow.

12. Acceptance Criteria

  1. DataModelForm accepts steppedForm prop and renders one section at a time when true
  2. Step indicator shows completed (green check), active (blue number), and pending (gray) states
  3. Tapping a completed step dot navigates back to that step
  4. "Next Step" validates current section's required fields before advancing
  5. "Previous Step" navigates back without validation, preserving all state
  6. Summary step auto-generated as final step, rendering the form in disabled mode (all sections visible, all inputs read-only)
  7. Summary step edit pencils navigate to the corresponding section step
  8. After editing a section from Summary, navigating forward returns to Summary
  9. Custom form controls render in disabled state (no-op setFormData) in the Summary step
  10. Flyout Cancel button shows discard confirmation when form is dirty
  11. "Submit" on Summary step creates the record, saves all data atomically, and sets status to Submitted
  12. Form state is preserved across step navigation (no data loss)
  13. steppedForm = false (default) renders standard form — no regression
  14. Work Order mobile create uses steppedForm with all 5 WO sections
