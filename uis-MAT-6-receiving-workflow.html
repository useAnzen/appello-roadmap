1 · Business Case & Customer Evidence ▼

Without receiving, there is no proof of delivery, no way to dispute short shipments, and no automated invoice approval. Accountants manually compare every supplier invoice against orders — a process that breaks at scale. Receiving closes the loop between procurement and accounts payable.

Customer Evidence

"Match packing slip / bill of lading against purchase order. Mark full receipt or partial receipt. Handle back-orders." — Product vision (Nov 2025)
"If a vendor has a credit, it starts with receiving — you can't know what's wrong until you confirm what arrived." — Corey Shelson

Value Delivered

📦
Delivery Confirmation
Formal record of what arrived, when, and who received it — evidence for disputes and audits.
📋
Partial Receipts
Receive what arrived, track what’s back-ordered. Multiple receipts per PO until fully received.
🔀
Three-Way Match
PO qty vs Received qty vs Invoiced qty — instant visibility into discrepancies before paying.
🔄
PO Status Auto-Transition
PO status automatically moves through Partially Received → Fully Received as receipts are logged.

Competitive Landscape

HAS Procore Receiving against PO with partial receipt tracking
HAS Sage 300 Full receiving + AP matching with three-way match
HAS ServiceTitan Inventory receiving workflow
NONE Appello backOrderValue field only — no receiving workflow

2 · Scope ▼

In Scope

  • IN PurchaseOrderReceipt + ReceiptLineItem Prisma models
  • IN Extensions to PurchaseOrderLineItem (receivedQty, backOrderQty — computed from receipts)
  • IN Receiving form: select PO, enter received qty per line item
  • IN Three-way match view on PO detail (PO vs Received vs Invoiced)
  • IN PO status auto-transitions: PartiallyReceived, FullyReceived
  • IN Committed material costs in Job Financial Summary
  • IN Receipt history tab on PO detail page

Out of Scope

  • OUT Mobile receiving
  • OUT Barcode/QR scanning on receipt
  • OUT Supplier invoice matching (AP integration)
  • OUT Automatic inventory creation (MAT-7)

4 · Data Model ▼

PurchaseOrderReceipt

model PurchaseOrderReceipt {
  id              String    @id @db.VarChar(21)
  isDeleted       Boolean   @default(false)
  createdAt       DateTime  @default(now())
  updatedAt       DateTime  @default(now()) @updatedAt
  purchaseOrderId String    @db.VarChar(21)
  PurchaseOrder   PurchaseOrder @relation(fields: [purchaseOrderId], references: [id])
  receivedById    String    @db.VarChar(21)
  receivedDate    DateTime  @default(now())
  notes           String?   @db.Text
  LineItems       ReceiptLineItem[]
  @@index([purchaseOrderId, isDeleted])
}

ReceiptLineItem

model ReceiptLineItem {
  id                      String    @id @db.VarChar(21)
  isDeleted               Boolean   @default(false)
  createdAt               DateTime  @default(now())
  updatedAt               DateTime  @default(now()) @updatedAt
  receiptId               String    @db.VarChar(21)
  Receipt                 PurchaseOrderReceipt @relation(fields: [receiptId], references: [id])
  purchaseOrderLineItemId String    @db.VarChar(21)
  PurchaseOrderLineItem   PurchaseOrderLineItem @relation(fields: [purchaseOrderLineItemId], references: [id])
  quantityReceived        Decimal   @db.Decimal(10, 2)
  quantityDamaged         Decimal?  @db.Decimal(10, 2)
  notes                   String?   @db.Text
  @@index([receiptId, isDeleted])
}

PurchaseOrderLineItem Extensions

FieldTypeNotes
receivedQtyComputedSum of ReceiptLineItem.quantityReceived across all receipts for this line
backOrderQtyComputedquantity - receivedQty — remaining quantity not yet received

Both receivedQty and backOrderQty are computed at query time from receipt data — not stored as columns. This avoids drift between receipts and summary fields.

5 · API Surface ▼

GraphQL Types

type PurchaseOrderReceipt {
  id: ID!
  purchaseOrderId: String!
  PurchaseOrder: PurchaseOrder!
  receivedById: String!
  ReceivedBy: User!
  receivedDate: DateTime!
  notes: String
  LineItems: [ReceiptLineItem!]!
  createdAt: DateTime!
}

type ReceiptLineItem {
  id: ID!
  receiptId: String!
  purchaseOrderLineItemId: String!
  PurchaseOrderLineItem: PurchaseOrderLineItem!
  quantityReceived: Float!
  quantityDamaged: Float
  notes: String
}

extend type PurchaseOrderLineItem {
  receivedQty: Float!
  backOrderQty: Float!
  Receipts: [ReceiptLineItem!]!
}

Input Types

input ReceiveItemInput {
  purchaseOrderLineItemId: String!
  quantityReceived: Float!
  quantityDamaged: Float
  notes: String
}

input ReceivePurchaseOrderInput {
  purchaseOrderId: String!
  receivedDate: DateTime
  notes: String
  lineItems: [ReceiveItemInput!]!
}

Queries & Mutations

type Query {
  purchaseOrderReceipts(purchaseOrderId: String!): [PurchaseOrderReceipt!]!
  purchaseOrderReceipt(id: ID!): PurchaseOrderReceipt!
}

type Mutation {
  receivePurchaseOrder(input: ReceivePurchaseOrderInput!): PurchaseOrderReceipt!
}

The receivePurchaseOrder mutation validates that quantityReceived ≤ remainingQty per line, creates receipt + line item records in a transaction, and auto-transitions PO status to PartiallyReceived or FullyReceived.

7 · Acceptance Criteria ▼

  • A full quantity can be received against a Supplier PO in a single receipt
  • Partial quantities can be received, leaving a back-order balance on remaining lines
  • Back-order quantity is correctly computed as ordered - receivedQty
  • PO status auto-transitions to PartiallyReceived when some (but not all) lines are fully received
  • PO status auto-transitions to FullyReceived when all lines reach ordered quantity
  • Three-way match view displays PO qty, received qty, and invoiced qty per line with status indicators
  • Three-way match highlights discrepancies (invoiced ≠ received) with red indicator
  • Receipt history tab on PO detail shows all receipts with date, receiver, item count, and notes
  • Permission check: materialorder.edit required to receive against a PO
  • Validation: received quantity cannot exceed remaining quantity (ordered - previouslyReceived)
  • Committed material costs from issued POs appear in the Job Financial Summary
  • Damaged quantity can be recorded per line item during receiving
