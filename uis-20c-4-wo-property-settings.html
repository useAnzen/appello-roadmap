WO-3d Work Orders Draft

Work Order Property Settings

Add a "Work Orders" tab to Property Settings with its own layout builder, default properties (including WO statuses), and form builders. Handle the dual-status pattern where jobStatusId references either jobStatus or workOrderStatus PropertyValues depending on the job's classification.

Sequence: WO-3d Estimate: 2-3 days Requires: WO-1 (JobClassification), WO-3a (DAA setup with WorkOrder subModel)

1. The Dual-Status Problem

Job status is stored in Job.jobStatusId (FK to PropertyValue). Currently, all status options come from propertyName = "jobStatus". Work Orders need a different status set (Draft, Submitted, Awaiting Signature, Signed, Approved, Invoiced, Void) seeded as propertyName = "workOrderStatus" in WO-1.

Same database field, different PropertyValue sets:

PropertyValue propertyName"jobStatus""workOrderStatus"

This is the same pattern as having different status sets for different entity types sharing a model. The resolution is classification-aware: when the form/detail page knows the job is a WorkOrder, it queries workOrderStatus PropertyValues for the status dropdown. When it's a ProjectJob, it queries jobStatus.

2. New "Work Orders" Tab in Property Settings

Add a new top-level tab in PropertySettings.tsx alongside the existing "Jobs" tab. This gives admins a dedicated place to configure WO-specific layouts, properties, and statuses.

What the tab contains

Uses the same DataModelSettingsTabs component as Companies, Contacts, etc. — but with subModel passed to scope everything to WorkOrder DAAs and sections.

3. Classification-Aware Status Input

The jobStatusId field on the Job form needs to show the correct status options depending on the job's classification. This is handled at the form level, not the model level.

How it works

  • The job-status DAA exists in both the ProjectJob and WorkOrder DAA sets (duplicated in WO-3a)
  • The form component for the status field checks the job's classification (from the selected jobTypeId) to determine which propertyName to query for options
  • ProjectJob: queries PropertyValues where propertyName = "jobStatus"
  • WorkOrder: queries PropertyValues where propertyName = "workOrderStatus"
  • The customFormControl for the status DAA slug implements this branching
// In the Job form's customFormControl or renderChild:
case "job-status":
    const statusPropertyName = jobClassification === "WorkOrder"
        ? "workOrderStatus"
        : "jobStatus";

    return (
        <PropertyValueSelector
            propertyName={statusPropertyName}
            value={formData.jobStatusId}
            onChange={(id) => setFormData({ ...formData, jobStatusId: id })}
            label={assignment.defaultAttributeLabel}
        />
    );

Also applies to:

  • Detail page: Status badge resolves the display name from the correct PropertyValue set
  • List page filters: Work Orders list status filter queries workOrderStatus; Jobs list queries jobStatus
  • Table feed: Status column display resolves via the PropertyValue relation (already works since jobStatusId is a direct FK — the PropertyValue knows its own label)
No schema changes needed for this. The jobStatusId FK on Job just points to a PropertyValue. Whether that PropertyValue has propertyName = "jobStatus" or "workOrderStatus" doesn't matter at the data level — the FK resolves either way. The classification-awareness is purely a UI concern: which options to show in the dropdown.

4. Scoping Layout Builders by subModel

The DocumentSectionLayoutBuilder and DataModelFormBuilderWrapper in the WO settings tab need to only show/edit WorkOrder sections and DAAs, not ProjectJob ones.

Current state

The existing JobSettingsTabs passes initialReferenceModel="Job" to DocumentSectionLayoutBuilder without a subModel. After WO-3a's backfill, all Job DAAs have a subModel set, so the layout builder needs to be told which set to show.

Changes needed

  • DocumentSectionLayoutBuilder: Accept an optional subModel prop. When provided, only show sections and DAAs matching that subModel. When not provided, show all (backward compat for models without subModel like Company, Contact, etc.)
  • DataModelFormBuilderWrapper: Same — accept optional subModel prop for scoping.
  • JobSettingsTabs: Update to pass subModel="ProjectJob" to its layout builders
  • WorkOrderSettingsTabs (new): Pass subModel="WorkOrder" to its layout builders

PropertyValues filtering

The Default Properties sub-tab uses PropertyValues with referencedOn={["Job"]}. For the WO tab, we need to either:

  • Option A: Pass a filter for propertyName to only show WO-related PropertyValues (e.g., workOrderStatus, jobType where classification = WorkOrder)
  • Option B: Use a different referencedOn value (e.g., ["WorkOrder"]) and update the WO status PropertyValues to have referencedOn = "WorkOrder"

Recommendation: Option A — filter by propertyName. The PropertyValues component already supports query filters. This avoids changing the PropertyValue model.

7. Acceptance Criteria

  1. "Work Orders" tab visible in Property Settings dropdown (after "Jobs")
  2. Work Orders settings tab shows: Detail View Layout, Settings View Layout, Manage Forms, Custom Properties, Default Properties
  3. Layout builders in WO settings only show WorkOrder sections and DAAs (not ProjectJob)
  4. Layout builders in Jobs settings only show ProjectJob sections and DAAs (not WorkOrder)
  5. Default Properties in WO settings shows workOrderStatus PropertyValues
  6. Default Properties in Jobs settings continues to show jobStatus PropertyValues
  7. WO form status dropdown shows WO statuses (Draft, Submitted, etc.)
  8. Project Job form status dropdown shows job statuses (Bid, Active, etc.)
  9. Both status sets write to the same jobStatusId field on Job
  10. Status badge on WO detail/list pages resolves correctly from the PropertyValue
  11. Work Orders list status filter shows WO statuses only
  12. Jobs list status filter shows job statuses only
  13. Admins can add/edit/reorder WO statuses independently from job statuses
