1 · Business Case & Customer Evidence ▼

The create/edit form is the primary interaction point for T&M work orders. It mirrors the existing JobAddEditForm pattern (flyout-based, section-by-section) but adds structured line items, auto-pricing from rate tables, and on-device signature capture. PMs create work orders from desktop; field workers will create from mobile (WO-8).

Customer Evidence

"We need a single form where the PM can fill in the job info, add labor and material lines, and capture a customer signature — all in one flow, not bouncing between screens." — Tear Systems (UX Workshop)
"Draft saving is critical. Our guys start a work order in the morning, add lines throughout the day, and submit at end of shift." — AB Insulation (Field Observation)
📝
Single-Flow Creation
All 8 form sections in one flyout — no multi-page wizard or tab switching required.
💾
Draft Save
Save incomplete work orders as Draft status, return and complete later.
🔢
Auto-Sequenced Numbers
WO number auto-generated on create (WO-001, WO-002, ...) — no manual entry.
✍️
Signature Capture
Canvas-based signature pad with GPS stamping, stored as asset attachments.

2 · Scope ▼

In Scope — 8 Form Sections

  1. IN Work Order Info — Name, WO number (auto), date of work, customer reference, status
  2. IN Company & Customer — Company selector (AssignViewButton), contact person
  3. IN Labor Items — Repeating-row editor (from WO-2 LaborItemEditor)
  4. IN Material Items — Repeating-row editor (from WO-2 MaterialItemEditor)
  5. IN Equipment Items — Repeating-row editor (from WO-2 EquipmentItemEditor)
  6. IN Summary & Pricing — Auto-calculated subtotals, markup %, overhead %, tax %, grand total
  7. IN Photos — Photo upload via existing asset system
  8. IN Signatures — Canvas signature pad with GPS capture

Additional In-Scope

  • IN Draft save functionality (save without validation)
  • IN Auto-sequenced WO number via generateWorkOrderNumber
  • IN Full vertical flyout form (matches JobAddEditForm pattern)

Out of Scope

  • OUT Offline form drafting (Phase 2 — requires service worker)
  • OUT Voice-to-text entry for description fields
  • OUT Template-based pre-fill (Phase 2)

4 · Data Model ▼

The form consumes the JobClassification system from WO-1 and line item models from WO-2. This spec adds WO-specific fields to the Job model and defines the revised WorkOrderLaborItem model.

Origin: These fields were identified during the ClearStory competitive analysis and customer requirements workshops. They were originally scoped in WO-1 but moved here since WO-1 now focuses on classification/routing only. These fields are consumed by the WO create/edit form, detail page (WO-4), and mobile creation (WO-8).

WorkOrderLaborItem Model (Revised from WO-2)

Key changes from original WO-2 spec: Removed instanceId (only 1 instance), removed straightTimeHours/overtimeHours/doubleTimeHours/rate/overtimeRate/doubleTimeRate (all derived from TradeLevelRate). Replaced tradeLevelId with tradeLevelRateId to capture the exact rate record. Added qtyOfWorkers. Duration stored in seconds matching the timesheet pattern. 
model WorkOrderLaborItem {
  id                    String              @id @db.VarChar(21)
  isDeleted             Boolean             @default(false)
  isArchived            Boolean             @default(false)
  createdAt             DateTime            @default(now())
  updatedAt             DateTime            @default(now()) @updatedAt

  jobId                 String              @db.VarChar(21)
  Job                   Job                 @relation(fields: [jobId], references: [id])

  tradeLevelRateId      String              @db.VarChar(21)
  TradeLevelRate        TradeLevelRate      @relation(fields: [tradeLevelRateId], references: [id])

  qtyOfWorkers          Int                 @default(1)
  submittedDuration     Int                 @default(0)  // seconds
  description           String?             @db.VarChar(500)
  orderNo               Int                 @default(0)

  @@index([jobId, isDeleted])
}

How one visual row becomes multiple records

In the UI, one labor row shows: Employee Agreement → Trade Level → rate label columns (Wages 1x, 1.5x, 2x, etc.) → Qty of Workers. In the database, each rate label with hours > 0 is a separate WorkOrderLaborItem record, all sharing the same orderNo.

UI RowDatabase Records
Journeyman — 3 workers — 8.0 hrs ST, 2.0 hrs OT Record 1: tradeLevelRateId = Journeyman/Wages 1x, qtyOfWorkers = 3, submittedDuration = 28800 (8 hrs), orderNo = 0
Record 2: tradeLevelRateId = Journeyman/Wages 1.5x, qtyOfWorkers = 3, submittedDuration = 7200 (2 hrs), orderNo = 0
Foreman — 1 worker — 8.0 hrs ST Record 3: tradeLevelRateId = Foreman/Wages 1x, qtyOfWorkers = 1, submittedDuration = 28800, orderNo = 1

tradeLevelRateId encodes everything

TradeLevelRate already links to:

  • EmployeeAgreementContract → EmployeeAgreement (which agreement & contract version)
  • TradeLevel (which classification — Journeyman, Foreman, etc.)
  • EmployeeAgreementRateLabel (which rate column — Wages 1x, 1.5x, 2x, etc.)

From tradeLevelRateId you can derive: baseValue (cost), billableValue (what the client pays, from LEM-1), the agreement name, the trade level name, and the rate label name. No redundant fields needed on the line item.

Contract selection based on Work Order date

The TradeLevelRate used must come from the EmployeeAgreementContract that is active for the WO's dateOfWork. When loading the form:

// Find the active contract for the WO date:
const activeContract = await prisma.employeeAgreementContract.findFirst({
    where: {
        employeeAgreementId: agreement.id,
        effectiveDate: { lte: workOrder.dateOfWork },
        isArchived: false,
        isDeleted: false
    },
    orderBy: { effectiveDate: "desc" }
});
// Then load TradeLevelRates for that contract

This ensures the rates used on the WO match the contract that was active when the work was performed, not the current contract (which may have different rates).

Employee Agreement auto-selection

The form auto-selects the EmployeeAgreement associated with the user filling out the WO (via the user's employment record). If the user is linked to multiple agreements, a selector appears. If only one agreement exists, it auto-selects and the field shows as read-only with "(auto)" label.

// Labor items state — two types: structured (trade level) and blank (manual)
interface IStructuredLaborRow {
    type: "structured";
    orderNo: number;
    employeeAgreementId: string;      // auto from submitter
    tradeLevelId: string;             // selected per row
    qtyOfWorkers: number;
    rateLabelDurations: Record<string, number>;  // rateLabelId → seconds
    description?: string;
}
interface IBlankLaborRow {
    type: "blank";
    orderNo: number;
    description: string;              // required on blank lines
    qtyOfWorkers: number;
    submittedDuration: number;        // seconds
    rate: number;                     // manual rate entry
}
type ILaborRow = IStructuredLaborRow | IBlankLaborRow;
const [laborRows, setLaborRows] = useState<ILaborRow[]>([]);

5 · API Surface ▼

This slice consumes existing mutations — no new API endpoints are introduced.

Consumed Mutations

MutationSourcePurpose
createJobExisting + WO-3Creates the work order with a WORK_ORDER-classified jobType. WO-3 adds the WO-specific fields (dateOfWork, markupPercent, etc.).
editJobExisting + WO-3Updates work order fields (name, dates, percentages)
upsertWorkOrderLineItemsWO-2Batch saves labor, material, and equipment items
recalculateWorkOrderTotalsWO-2Recomputes subtotals and grand total after line item changes
createAssetExistingUploads photos and signature images

Form Submission Flow

1. createJob({ jobTypeId: [WO-classified type], name, dateOfWork, ... }) → jobId
2. upsertWorkOrderLineItems({ jobId, laborItems, materialItems, equipmentItems })
3. createAsset() × N  (for each photo and signature)
4. recalculateWorkOrderTotals({ jobId })
5. If status !== "Draft": editJob({ status: "Submitted" })

7 · Acceptance Criteria ▼

  • The flyout form opens with all 8 sections visible and scrollable
  • WO number is auto-generated on form open (e.g. "WO-003") and displayed as read-only
  • Company can be selected via AssignViewButton and displays the selected company name
  • Labor, Material, and Equipment item editors render inline and support add/edit/remove rows
  • Summary section auto-calculates subtotals and grand total as line items are added/modified
  • "Save Draft" saves the work order with status "Draft" without enforcing required field validation
  • "Submit" validates all required fields and sets status to "Submitted"
  • Photos can be uploaded via drag-and-drop or file browser and are stored as assets
  • Signature pad captures a drawn signature, stores it as an asset, and records signer name + GPS coordinates
