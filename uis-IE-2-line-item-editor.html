Itemized Line Item Editor

Hierarchical line-item editor (sections + subsections + priced lines) inside EstimateChangeOrderDetails, mounted only when creationType = Itemized. Blank rows start with a Type dropdown; picking a type reveals a type-specific progressive cascade that resolves a rate-level row (or falls back to a Custom free-text line). Per-line stacked Tax and Discount sub-rows, mirroring SimpleInvoice.tsx (apps/frontend/src/components/DetailsPage/JobDetailsPage/Tabs/Billing/SimpleInvoice.tsx lines 335–491). Data contract in IE-1 field matrix; discount data model in SD-1.

1. Business Case ▼

Estimators need the same mental model as field T&M capture: pick catalog rows or type free-form descriptions, see unit cost populate, adjust quantity and markup.

Combination of dropdown picks and manual dollar entry on the same grid. — Clement / product direction (April 2026)

2. Scope ▼

In Scope

  • IN EstimateLineItemEditor.tsx + row / section / cascade sub-components. Mounts only when EstimateChangeOrderRecord.creationType = Itemized — no mode toggle.
  • IN Blank-row Type dropdown (Labor / Material / Equipment / Custom) with type-specific progressive cascades: Labor (TradeLevel → EmployeeAgreementRateLabel, + optional worker), Material (Material → X → Y → Z-when-defined), Equipment (Equipment → Per → Rate Type → Operator Included), Custom (free-text description)
  • IN Each completed cascade resolves a single rate-level row (TradeLevelRate / MaterialAttributePrice / EquipmentRateValue) and calls IE-1's resolveItemizedLineItemUnitCost to seed unit + unitCost. Catalog-linked rows render unit / unitCost / totalCost / price as read-only.
  • IN Per-line stacked Tax Rate(s) sub-row (dropdown + amount + remove per entry, with “+ Add Another Tax Rate”); consumes IE-1's M2M TaxType relation. Mirrors the pattern in apps/frontend/src/components/DetailsPage/JobDetailsPage/Tabs/Billing/SimpleInvoice.tsx (lines 335–491).
  • IN Per-line stacked Discount(s) sub-row: picker offers SD-1 suggestedDiscountsForLine (Job ∪ Company, de-duped) + an Ad-hoc entry option (% / $ toggle + value). Wires IE-1's line to SD-1's attachDiscount / detachDiscount / createAdHocDiscountAndAttach.
  • IN Estimate-level discount stack in the totals strip (same sub-row UX; referenceModel = "EstimateChangeOrderRecord"; orderNo drives application order). The totals strip (rendered by IE-3's EstimateChangeOrderTotals) MUST surface line discounts subtotal and estimate-level discount subtotal as two distinct rows — never merged — so the reader can always tell which level the money came from. Tax is always labeled as a sum-from-lines rollup (weighted average of per-line rates); there is no estimate-level tax control.
  • IN Seed a new priced line's markupPercent from ECR.estimateMarkup at insert time. The field is semantically overloaded by design: on Full / BidFirst it's an applied multiplier; on Itemized it's a per-line seed that's never applied directly to totals (line-level markupPercent drives sell). Changing the seed later does not retroactively re-price existing lines — that's a deliberate bulk action via IE-3's “Set default markup %” modal.
  • IN Nested section tree — per-section + Sub / + Line / ↑ ↓ / delete controls; root-level + Section / + Line on the worksheet toolbar; depth cap 3 enforced against IE-1's MAX_TREE_DEPTH
  • IN Wire CRUD (create / createMany / edit / editMany / delete / deleteMany / archive / restore) to IE-1's GraphQL surface (per-row now; batched upsert once IE-3 extends the parent mutation)
  • IN Client-side filter (search box) across section titles and line descriptions

Out of Scope

  • OUT Discount / DiscountAssignment data model — SD-1
  • OUT ItemizedEstimateChangeOrderRecordLineItem data model, enums, validators — IE-1
  • OUT Itemized creationType enum addition and the creation-type gate — IE-3
  • OUT Totals math (line % → line sell → estimate-level on pre-tax → proportional tax split), overhead %, gross margin, contingency — IE-3
  • OUT PDF / invoice conversion — IE-4, IE-5

3. UI Concepts ▼

A hierarchical grid of sections, subsections, and priced line items. New lines are added via a blank-row Type dropdown followed by a type-specific progressive cascade that resolves a rate-level row (TradeLevelRate / MaterialAttributePrice / EquipmentRateValue) or falls back to a free-text Custom row. Each priced row pairs with right-aligned stacked sub-rows for Tax Rate(s) and Discount(s); the totals strip below the grid holds the estimate-level discount stack.

3.2 Add-line flow (progressive dropdowns)

Clicking + Add item inserts a blank row with a Type dropdown (Labor / Material / Equipment / Custom). Once the type is picked, additional dropdowns cascade on the same row until enough selections nail down a specific rate-level row — at which point the resolver fires and the unit / unitCost back-fill, and markupPercent seeds from ECR.estimateMarkup (the same Instance default that feeds Full / BidFirst estimates). The final catalog cascade determines priceReferenceModel + priceReferenceId (see IE-1 §3). No unified search across types — the type picker disambiguates up front.

Labor Labor
Cascade: Type → TradeLevel → EmployeeAgreementRateLabel
1. Pick TradeLevel.
2. Pick EmployeeAgreementRateLabel (filtered to labels active on the chosen trade + the default employee agreement).
3. The pair resolves a unique TradeLevelRate; its billableValue seeds unitCost, unit defaults to hr.
4. Optional userId slot on the row (Labor only, per IE-1) — pick a worker later if you want this specific person's agreement applied; otherwise the contract base rate stands.
5. priceReferenceModel = TradeLevelRate, priceReferenceId = <TradeLevelRate.id>.
Material Mat
Cascade: Type → Material → X attribute → Y attribute (→ Z attribute if material defines a third axis)
1. Pick Material.
2. Pick X attribute value; options filtered to values present on that material's MaterialAttributePrice rows.
3. Pick Y attribute value; options filtered by the X pick.
4. (Optional) Z attribute when the material defines one — resolves the final MaterialAttributePrice row.
5. unit = material UOM; unitCost = MaterialAttributePrice.value for the chosen (X, Y, Z) cell.
6. priceReferenceModel = MaterialAttributePrice, priceReferenceId = <MaterialAttributePrice.id>.
Equipment Eq
Cascade: Type → Equipment → Per (EquipmentRatePer) → Rate Type (EquipmentRateLabel) → Operator Included
1. Pick Equipment.
2. Pick Per billing period (Hour / Day / Week / Month / Year) — options from active EquipmentRateSchedule for the equipment's category.
3. Pick Rate Type (EquipmentRateLabel — e.g. Standard, Prevailing Wage, After Hours).
4. Operator Included toggle (drives the with/without-operator rate split on EquipmentRateValue when present).
5. The four picks resolve a unique EquipmentRateValue; baseValue seeds unitCost, unit derives from the Per choice.
6. priceReferenceModel = EquipmentRateValue, priceReferenceId = <EquipmentRateValue.id>.
Custom Custom
Cascade: Type → free-text description
1. Pick Custom → blank row.
2. description, quantity, unit, unitCost all manual.
3. priceReferenceModel and priceReferenceId remain null.
4. No resolver call.

Unit cost becomes editable for override once the resolver seeds it (all four types). The cascade dropdowns on the row are re-openable until the row is saved; once saved, swapping the catalog link is governed by the same rule as lineItemType — delete and re-add (see §3.3).

5. Data & API ▼

Consumed from IE-1 (line items)

  • itemizedEstimateChangeOrderRecordLineItems(options) — listing
  • resolveItemizedLineItemUnitCost(input) — auto-fills unit + unitCost after each cascade completes
  • createItem / createMany / editItem / editMany / deleteItem / deleteMany / archive / restore family — full CRUD including section nodes (nodeType = Section) and line nodes
  • nodeType and lineItemType are canEdit: false on the model — edit payloads that change them are stripped. UI enforces delete-and-re-add

Consumed from SD-1 (discounts)

  • suggestedDiscountsForLine(lineItemId) — populates the per-line Discount picker (Job + Company union)
  • discountsForEntity(referenceModel, referenceId) — read path for both line and estimate-level stacks
  • attachDiscount(input) — attach a reusable discount from the picker
  • createAdHocDiscountAndAttach(input) — single-call ad-hoc path for inline “type 5%” entries
  • detachDiscount(id) — soft-delete a line- or estimate-level assignment
  • reorderDiscountAssignments(…) — drag-reorder within a stack

Optional until IE-3: upsertItemizedEstimateChangeOrderRecordLineItems(estimateId, items[]) for transactional save of the whole tree with header totals in a single mutation.

7. Acceptance Criteria ▼

  • Editor mounts only when creationType = Itemized; no worksheet/itemized toggle is rendered in either direction
  • User can add, edit, remove rows without full page reload
  • Clicking + Add item inserts a blank row with a Type dropdown; subsequent cells stay disabled until Type is picked
  • Labor cascade (TradeLevel → EmployeeAgreementRateLabel) resolves to a single TradeLevelRate and seeds unitCost from billableValue
  • Material cascade (Material → X → Y → Z when defined) resolves to a single MaterialAttributePrice and seeds unitCost from value
  • Equipment cascade (Equipment → Per → RateLabel → Operator) resolves to a single EquipmentRateValue and seeds unitCost from baseValue
  • Custom row permits manual description / quantity / unit / unitCost; priceReferenceModel and priceReferenceId stay null
  • On Labor / Material / Equipment rows, unit, unitCost, totalCost, and price render as read-only text — no inline input. Any correction to the rate happens at the source (TradeLevelRate / MaterialAttributePrice / EquipmentRateValue) and flows back in on the next resolver run
  • Labor rows expose an optional userId picker (Labor only — never required)
  • Type badge and catalog link are locked after first save; UI offers delete + re-add to change
  • Tax Rate(s) sub-row stacks multiple TaxType entries per line with a dropdown + amount + remove per entry, plus + Add Another Tax Rate, matching SimpleInvoice.tsx UX
  • Discount(s) sub-row stacks multiple DiscountAssignment entries per line with the same pattern; suggested entries (from Job / Company) + ad-hoc entry option
  • Adding / editing / removing a line-level discount or tax triggers recompute of taxValue on the line
  • Estimate-level discount stack lives in the totals strip (same sub-row UX, referenceModel = "EstimateChangeOrderRecord")
  • Section rows render + Sub and + Line buttons inline (no “Here” indirection); root-level + Section / + Line live on the editor toolbar
  • Section reorder moves the entire subtree; depth cap enforced (rejects + Sub when it would exceed IE-1's cap)
  • Total cost, markup fields, line price, tax total, and discount total stay numerically consistent after every change
  • Permissions match existing estimate edit rules
