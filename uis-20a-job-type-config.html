1 · Business Case & Customer Evidence ▼

This is the foundation work item for the entire T&M Work Orders feature. Without the isWorkOrder flag, WO-specific status workflows, and job-type form configurations, no work order can be created, submitted, or invoiced. Every downstream slice (WO-2 through WO-8) depends on this slice being complete.

Customer Evidence

"We need to be able to create different fields per job type. Our T&M work orders need completely different information than our project-based jobs." — Tear Systems (Discovery Call)
"Each template should have its own form — the fields for insulation work orders are nothing like the fields for fireproofing." — AB Insulation (Requirements Workshop)

2 · Scope ▼

In Scope

  • IN isWorkOrder Boolean flag on Job model with @default(false)
  • IN PropertyValue seeds for 7 WO statuses: Draft, Submitted, Awaiting Signature, Signed, Approved, Invoiced, Void
  • IN New jobType PropertyValue: "T&M Work Order"
  • IN DataModel form configuration for WO-specific fields
  • IN Job list filter toggle to separate Work Orders from Project Jobs
  • IN Auto-sequenced workOrderNumber per instance
  • IN WO-specific fields: dateOfWork, customerReferenceNumber, markup/overhead/tax percentages, signature fields

Out of Scope

  • OUT Custom status workflow editor (Phase 2)
  • OUT Per-customer form templates (Phase 2)
  • OUT Status transition permission controls

7 · Acceptance Criteria ▼

  • A new Job can be created with isWorkOrder: true and is stored with the flag in the database
  • The 7 WO statuses (Draft, Submitted, Awaiting Signature, Signed, Approved, Invoiced, Void) are seeded as PropertyValues and returned by the workOrderStatuses query
  • "T&M Work Order" appears as a selectable job type in the create job form
  • Selecting "T&M Work Order" renders WO-specific form fields (dateOfWork, customerReferenceNumber, markup/overhead/tax percentages)
  • The job list page shows a ButtonGroup filter (All Jobs | Project Jobs | Work Orders) that correctly filters by isWorkOrder
  • workOrderNumber auto-increments per instance and fullWorkOrderNumber formats as "WO-001", "WO-002", etc.
  • Existing project-based jobs are unaffected — isWorkOrder defaults to false for all existing records
