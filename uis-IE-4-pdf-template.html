Itemized Estimate PDF

Branch the existing estimate PDF pipeline when creationType = Itemized: reuse the current letterhead, merge-tag letter, totals block, and signature footer, and swap only the breakdown area for the itemized section / line tree. Client-facing only — markup %, markup $, and unit cost are never rendered.

1. Business Case ▼

Customers expect a printable proposal that mirrors the itemized breakdown they’ll see discussed on-screen — sectioned, priced at the line level, with tax and any discounts applied transparently. This PDF is the external approval artifact, so it must never expose internal margin data (unit cost, markup %, markup $).

2. Scope ▼

In Scope

  • IN Branch EstimateChangeOrderPDF.tsx to render an itemized breakdown area when creationType = Itemized. All other PDF chrome (logo, header, recipient block, merge-tag letter, terms, signature block, footer) is reused unchanged.
  • IN Itemized breakdown section: numbered section / subsection rows, line rows grouped under their section, stacked tax sub-rows + stacked discount sub-rows per line, section subtotals, estimate-level totals block.
  • IN Columns: Description, Qty, Unit, Unit price, Line total. No toggles.
  • IN Totals block sourced from the same itemizedTotalsEngine (IE-3). Rows (always rendered in order, even at $0.00 so the breakdown stays legible): line sell before discounts → line discounts subtotal (sum of per-line) → line sell after line discounts → estimate-level discount subtotal (stack total, distinct from line discounts) → overhead → tax subtotal (sum of per-line cached taxValue, labeled as weighted average) → grand total.
  • IN Wire pdf-export.ts to pass creationType + the itemized tree/totals payload into the PDF component.

Out of Scope

  • OUT Any “internal” or costed PDF variant. Cost, markup %, and markup $ are never rendered on this PDF.
  • OUT Per-export / per-estimate column visibility toggles. The itemized layout is fixed.
  • OUT Invoice PDF (existing Job bill PDFs).
  • OUT Worksheet / BidFirst / Full PDFs — those continue to use the existing pipeline untouched.

3. PDF Layout ▼

Only the breakdown area changes. Everything else on the PDF — letterhead, proposal header, recipient block, merge-tag letter, terms block, signature block, footer — is the existing EstimateChangeOrderPDF.tsx output reused verbatim. The itemized branch renders when creationType = Itemized; Full / BidFirst continue to hit the existing worksheet breakdown.

Column vocabulary aligns with the IE-2 editor and the UIS-IE glossary: Unit price = line price ÷ qty (already includes any per-line markup); Line total = line price. Cost, markup %, and markup $ are not columns on this PDF under any configuration.

3.1 Itemized breakdown area

Structure mirrors the editor grid from IE-2: section / subsection header rows (numbered via depth-first walk), line rows grouped beneath their parent section, stacked tax sub-rows and discount sub-rows indented beneath each line, and a section subtotal row. Section numbering is computed at render time — not persisted.

3.2 Full page composition (single page)

Chrome is reused from the existing PDF; only the breakdown area in the middle changes. Totals block at the bottom of the breakdown is sourced from the shared itemizedTotalsEngine (IE-3) — same engine the on-screen totals panel reads from, so PDF totals and screen totals cannot diverge.

3.3 Multi-page behavior

When the breakdown overflows the printable region: repeat the table header on each page; keep line + its tax / discount sub-rows together (never split a line from its sub-rows); the totals block and signature appear on the last page only. Page break handling is the same primitive used by the existing worksheet PDF — no new engine.

4. Change Surface ▼

  • MOD apps/frontend/src/components/DownloadablePDF/EstimateChangeOrder/EstimateChangeOrderPDF.tsx — add conditional branch: when creationType === Itemized, render the new itemized breakdown component in place of the worksheet breakdown. All surrounding chrome is unchanged.
  • NEW apps/frontend/src/components/DownloadablePDF/EstimateChangeOrder/ItemizedBreakdown.tsx — new component rendering section / subsection header rows, line rows with stacked tax & discount sub-rows, and section subtotals using @react-pdf primitives.
  • MOD apps/frontend/src/pages/api/pdf-export.ts — pass creationType + itemized tree + totals payload (from the shared itemizedTotalsEngine) into the PDF component.

5. Acceptance Criteria ▼

  • Itemized estimates render a PDF where the breakdown area matches the IE-2 editor: section / subsection headers with depth-first numbering, line rows grouped under their section, tax sub-rows + discount sub-rows stacked beneath each line, section subtotals.
  • Columns are exactly: Description, Qty, Unit, Unit price, Line total. No cost, markup %, or markup $ column appears under any configuration.
  • Letterhead, recipient block, merge-tag letter, terms, signature block, and footer are pixel-identical to the existing worksheet PDF (same source components).
  • Totals block is computed by the shared itemizedTotalsEngine (IE-3) — PDF totals match on-screen totals to the cent.
  • Totals block surfaces line discounts (sum of per-line discount amounts) and estimate-level discount (stack total) as two distinct rows, never merged. Each row always renders, even at $0.00.
  • Tax row is always labeled as a rollup from lines (sum of per-line taxValue) with a weighted-average rate annotation. There is no estimate-level tax line on the PDF; all tax originates from the line grid.
  • Multi-page: table header repeats, a line and its tax / discount sub-rows never split across pages, totals block + signature appear on the last page only.
  • Full / BidFirst estimates continue to use the existing worksheet PDF path with no regression.
