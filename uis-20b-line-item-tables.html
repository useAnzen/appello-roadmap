Work Order Line Item Tables

1 · Business Case & Customer Evidence ▼

Structured line items are what separates a first-class work order system from a form-based workaround. Without discrete labor, material, and equipment rows, there is no way to generate accurate invoices or feed into job P&L. The WO is a field document — it captures what was done (trade levels, hours, equipment, quantities). Rates are resolved from the existing rate infrastructure (TradeLevelRate for labor via LEM-1, EquipmentRateValue for equipment via LEM-2) at invoice generation time (WO-7), not on the WO form itself.

2 · Scope ▼

In Scope

  • IN 3 new Prisma models: WorkOrderLaborItem, WorkOrderMaterialItem, WorkOrderEquipmentItem
  • IN Auto-pricing service that pulls rates from TradeLevel, Material, and Equipment tables
  • IN Repeating-row editor components for each line item type
  • IN Row-level total calculation (hours × rate for each pay type)
  • IN Upsert mutation for batch save of all line items
  • IN Recalculate totals mutation to update Job subtotals and grandTotal

Out of Scope

  • OUT Drag-and-drop reordering (Phase 1 uses manual orderNo)
  • OUT Bulk import from external price sheets
  • OUT Custom column visibility per user

3 · UI Concepts ▼

All UI mockups for line item editors (labor, material, equipment) are in WO-3 (Create/Edit Form) since that spec owns the form interface. WO-2 defines the data models and API surface only.

Key UI patterns defined in WO-3:

  • Labor: Structured lines (EmployeeAgreement + Trade Level + rate label columns) and blank lines (free-text description + manual rate). "+ Add Labor" dropdown with both options.
  • Material: Structured lines (Material catalog lookup) and blank lines (free-text description + manual qty/cost). Same dropdown pattern.
  • Equipment: Structured lines (Equipment + rate label + per) and blank lines (free-text description + manual qty/rate). Same dropdown pattern.

All three item types support both structured (auto-populated from rate tables) and blank (fully manual) line entries. No rates or dollar totals shown on the WO form for structured lines — pricing applied at invoice generation (WO-7).

4 · Data Model ▼

WorkOrderLaborItem (Revised)

Changes from original spec: Removed instanceId, straightTimeHours, overtimeHours, doubleTimeHours, rate, overtimeRate, doubleTimeRate, total. Replaced tradeLevelId with tradeLevelRateId. Added qtyOfWorkers and submittedDuration (seconds). One record per (orderNo + tradeLevelRateId) combination. Rate label columns with hours > 0 each become a separate record grouped by orderNo in the UI. 
model WorkOrderLaborItem {
  id                    String          @id @db.VarChar(21)
  isDeleted             Boolean         @default(false)
  isArchived            Boolean         @default(false)
  createdAt             DateTime        @default(now())
  updatedAt             DateTime        @default(now()) @updatedAt

  jobId                 String          @db.VarChar(21)
  Job                   Job             @relation(fields: [jobId], references: [id])

  // Structured line: tradeLevelRateId set, rate derived from TradeLevelRate
  tradeLevelRateId      String?         @db.VarChar(21)
  TradeLevelRate        TradeLevelRate? @relation(fields: [tradeLevelRateId], references: [id])

  // Blank line: tradeLevelRateId is null, user fills in description + rate manually
  rate                  Decimal?        @db.Decimal(10, 2)   // only used on blank lines

  qtyOfWorkers          Int             @default(1)
  submittedDuration     Int             @default(0)    // seconds
  description           String?         @db.VarChar(500)
  orderNo               Int             @default(0)

  @@index([jobId, isDeleted])
}

For structured lines: tradeLevelRateId encodes EmployeeAgreement + Contract (date-specific) + TradeLevel + RateLabel. Rates derived at invoice generation via LEM-1 billableValue. For blank lines: user enters description, hours, qty of workers, and a manual rate. Both types save as WorkOrderLaborItem records — the presence of tradeLevelRateId distinguishes them.

WorkOrderEquipmentItem (Revised)

Changes: Removed instanceId. Added structured line fields (equipmentRateLabelId, per, operatorIncluded, submittedDuration) and blank line field (unitRate). Supports both structured (equipment from catalog + LEM-2 rates) and blank (manual description + rate) entries, same pattern as labor. 
model WorkOrderEquipmentItem {
  id                      String              @id @db.VarChar(21)
  isDeleted               Boolean             @default(false)
  isArchived              Boolean             @default(false)
  createdAt               DateTime            @default(now())
  updatedAt               DateTime            @default(now()) @updatedAt

  jobId                   String              @db.VarChar(21)
  Job                     Job                 @relation(fields: [jobId], references: [id])

  // Structured line: equipmentId set, rates derived from LEM-2
  equipmentId             String?             @db.VarChar(21)
  Equipment               Equipment?          @relation(fields: [equipmentId], references: [id])
  equipmentRateLabelId    String?             @db.VarChar(21)
  EquipmentRateLabel      EquipmentRateLabel? @relation(fields: [equipmentRateLabelId], references: [id])
  per                     EquipmentRatePer?   @default(HOUR)
  operatorIncluded        Boolean             @default(true)
  isFullDayBilling        Boolean             @default(true)

  // Blank line: equipmentId is null, user fills in description + rate manually
  unitRate                Decimal?            @db.Decimal(10, 2)

  // Shared fields
  submittedDuration       Int                 @default(0)    // seconds
  quantity                Decimal?            @db.Decimal(10, 2)  // manual qty for blank lines
  description             String?             @db.VarChar(500)
  orderNo                 Int                 @default(0)

  @@index([jobId, isDeleted])
}

7 · Acceptance Criteria ▼

  • Labor items can be created with a TradeLevel selection and the rate, OT rate (1.5×), and DT rate (2×) are auto-populated
  • Material items can be created with a Material selection and the unit cost is auto-populated from the Material catalog
  • Equipment items can be created with an Equipment selection and the unit cost is auto-populated
  • Row-level totals calculate correctly: Labor = (ST × Rate) + (OT × OT Rate) + (DT × DT Rate); Material/Equipment = Quantity × Unit Cost
  • The upsertWorkOrderLineItems mutation saves all 3 item types in a single transaction
  • The recalculateWorkOrderTotals mutation updates the Job's laborSubtotal, materialSubtotal, equipmentSubtotal, and grandTotal
  • Deleting a line item (setting isDeleted: true) recalculates totals and removes the row from the UI
  • The repeating-row editor allows adding, editing, and removing rows without a page reload
