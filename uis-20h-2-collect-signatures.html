Collect Signatures

Two signature collection workflows for submitted work orders: Request Signature (send to internal Appello user for remote signing, mirrors form ACK pattern) and Collect In-Person (hand phone to external person — client, GC rep — who signs on the device without an Appello account). Both workflows can chain together on a single work order.

Estimate: 3-4 days Requires: WO-8a (stepped form + self-sign), WO-3a (Signature model), WO-4 (detail page)

1. Signature Workflows

After a work order is submitted (status = Submitted), the submitter and other authorized users can collect additional signatures via two methods. These are accessed from the WO detail page (WO-4), not the create form.

3. Workflow A: Request Signature (Internal)

Trigger

User taps "Request Signature" on the WO detail signatures section. Opens a user selection flyout.

User Selection

  • Standard CustomCombobox searching active Users in the organization
  • Shows user name + role
  • Can request from multiple users (one at a time, each creates a separate request)
  • Cannot request from a user who has already signed or has a pending request

On request

  1. Create a SignatureRequest record (new model — see Section 6) linking the WO to the requested user
  2. Send push notification via existing notification infrastructure: "[Requester Name] requested your signature on [WO Number]"
  3. Send in-app notification (activity feed entry)
  4. The request appears in the signatures section with status = REQUESTED

Recipient experience

  1. Receives push notification
  2. Tapping opens the WO detail page
  3. A banner at the top: "Your signature has been requested" with a "Sign Now" button
  4. "Sign Now" scrolls to / opens the signature area: canvas + GPS capture
  5. On sign, a Signature record is created with userId = the signing user, and the SignatureRequest status is updated to SIGNED

4. Workflow B: Collect In-Person (External)

Trigger

User taps "Collect In-Person" on the WO detail signatures section. Opens a dedicated signing view designed for device handoff.

Signing view flow

  1. WO read-only display: The full work order content is shown in a scrollable read-only view (same layout as the Summary step from WO-8a). The external person can review all details before signing.
  2. Signer info inputs: Below the WO content, the signer enters:
    • Name (required) — maps to Signature.signerName
    • Email (optional) — stored on the Signature for reference
    • Company (optional) — stored on the Signature for reference
  3. Signature canvas: Standard react-signature-canvas with Clear button
  4. GPS capture: On sign, capture browser geolocation (same as WO-3c)
  5. "Confirm Signature" button: Creates the Signature record and returns to WO detail
Device handoff UX: The signing view should feel like a self-contained document the external person is reviewing and signing. No app navigation is visible — just the WO content, signer info, and signature canvas. A simple "Back" in the header returns the device owner to the WO detail page (with a confirmation if the canvas has content).

5. Signature Record Handling

Both workflows create the same Signature model record (from WO-3a). The difference is which fields are populated.

Signature model additions: The Signature model from WO-3a needs two new optional fields: signerEmail String? @db.VarChar(191) and signerCompany String? @db.VarChar(191). These are useful for both tracking and potential future email-based signature workflows.

6. SignatureRequest Model

Tracks pending signature requests for Workflow A. This is a lightweight model that links a WO to a requested user and tracks whether they've signed.

model SignatureRequest {
    id              String    @id @db.VarChar(21)
    isDeleted       Boolean   @default(false)
    isArchived      Boolean   @default(false)
    createdAt       DateTime  @default(now())
    updatedAt       DateTime  @default(now()) @updatedAt
    referenceModel  String                         // "Job"
    referenceId     String    @db.VarChar(21)      // the WO's job ID
    requestedById   String    @db.VarChar(21)      // user who sent the request
    RequestedBy     User      @relation("SignatureRequestedBy", fields: [requestedById], references: [id])
    requestedUserId String    @db.VarChar(21)      // user being asked to sign
    RequestedUser   User      @relation("SignatureRequestedUser", fields: [requestedUserId], references: [id])
    status          String    @default("PENDING")  // PENDING | SIGNED | DECLINED
    signatureId     String?   @db.VarChar(21)      // FK to Signature once signed
    Signature       Signature? @relation(fields: [signatureId], references: [id])
    signedAt        DateTime?

    @@index([referenceModel, referenceId, isDeleted])
    @@index([requestedUserId, status, isDeleted])
}

Status transitions

TransitionTrigger
PENDING → SIGNEDRequested user signs on the WO
PENDING → DECLINEDRequested user declines (optional v1 — can just ignore)
Polymorphic pattern. SignatureRequest uses referenceModel + referenceId so it can be reused for other models that need signature requests in the future (e.g., Change Orders, Estimates). For now only used with referenceModel = "Job".

7. Notifications

Request Signature notification

Follows the existing notification infrastructure (FCM push + in-app activity feed). Same pattern as form acknowledgement notifications.

ChannelContent
Push (FCM)"[Requester Name] requested your signature on [WO Number]"
In-appActivity feed entry with link to WO detail page

Notification tap behavior

Opens the WO detail page. The "Your signature has been requested" banner is shown based on querying SignatureRequest records where requestedUserId = currentUser and status = PENDING.

Signature completed notification (back to requester)

ChannelContent
Push (FCM)"[Signer Name] signed [WO Number]"
In-appActivity feed entry

Sent to the user who created the SignatureRequest. Also applies to in-person signatures — the user who initiated "Collect In-Person" doesn't need a notification (they're present), but other users who previously signed or requested signatures may want visibility. This is a v2 consideration.

8. Signing View Component

A shared component used by both workflows. Shows the WO in read-only mode with a signature area at the bottom.

interface IWOSigningViewProps {
    jobId: string;
    mode: "internal" | "external";  // determines which fields to show
    signatureRequestId?: string;    // for Workflow A, to update request status
    onComplete: () => void;         // callback after signature saved
    onCancel: () => void;
}

Read-only WO content

Reuses the readOnlySummary renderers from WO-8a's Summary step. Each section displays its data using the same compact read-only format. This avoids building a second read-only view.

10. Acceptance Criteria

  1. Workflow A: "Request Signature" button on WO detail opens user picker flyout
  2. Selecting a user creates a SignatureRequest and sends push + in-app notification
  3. Requested user sees "Your signature has been requested" banner on WO detail
  4. "Sign Now" opens the signing view with signer name auto-populated from user record
  5. Signing creates a Signature with userId set and updates SignatureRequest status to SIGNED
  6. Requester receives a "signed" confirmation notification
  7. Workflow B: "Collect In-Person" button opens the signing view in external mode
  8. Signing view shows full WO content in read-only (reuses readOnlySummary renderers)
  9. External signer enters name (required), optionally email and company
  10. Signing creates a Signature with userId = null and signerName populated
  11. GPS coordinates captured on both workflows
  12. After signing, view returns to WO detail page
  13. Signatures section on WO detail shows all signatures (self, internal, external) with status badges
  14. Cannot request signature from user who already signed or has a pending request
  15. Chained flow works: self-sign → request internal → collect external on same WO
